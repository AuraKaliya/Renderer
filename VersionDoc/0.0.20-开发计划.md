# 0.0.20-开发计划

## 1. 版本目标

围绕参数化建模与选中系统的下一阶段演进，先将当前项目从“固定测试对象 + 对象级 descriptor”推进到“正式参数化对象 + 特征链 + 最小选中闭环”的第一版，使后续控制点、局部坐标和视口直接编辑具备稳定的数据模型基础。

本版本完成后，项目应至少具备：

- 正式的参数化对象模型
- 显式的对象 id / 特征 id
- 对象与特征分层的数据描述
- 测试界面中的对象列表与特征列表
- 非拾取式的最小选中、激活、添加、删除与启用/禁用闭环

## 2. 本版本范围

### 2.1 包含内容

- 将当前 `ParametricObjectDescriptor` 从“单对象求值描述”继续升级为更正式的对象级模型
- 为参数化对象引入稳定 id 与对象级元数据
- 为特征引入稳定 id 与特征级元数据
- 将当前：
  - `basePrimitive`
  - `operators`
  统一收口到“对象 + 特征链”的显式结构
- 在 `test_viewer` 中增加最小对象/特征层级 UI
- 建立最小 `SelectionState`
- 支持非拾取式的：
  - 选中对象
  - 激活对象
  - 选中特征
  - 激活特征
  - 添加特征
  - 删除特征
  - 启用 / 禁用特征

### 2.2 不包含内容

- 屏幕空间拾取
- 控制点模型
- 控制点拖拽
- gizmo
- mesh component 选择
- 框选与复杂多选
- 曲线 / 剖面驱动建模
- 完整对象树或历史栈编辑器

- 应先建立对象层、特征层与最小选中语义
- `active` 与 `selected` 必须分离建模
- 控制点与视口直接编辑应留到对象/特征语义稳定之后

## 4. 已确认的设计约束

### 4.1 参数化建模约束

- `parametric_model` 仍只负责 CPU 侧建模求值
- 不允许把 OpenGL / Qt / 视口交互细节暴露进 `parametric_model`
- 对象与特征的变化仍必须走统一链路：
  - descriptor change
  - rebuild
  - local bounds update
  - repository / GPU update
  - view strategy consumption

### 4.2 选中系统约束

- 本版本只做最小选中状态，不做屏幕拾取
- 必须显式区分：
  - `selected`
  - `active`
- 首批只要求单选稳定，多选不是当前版本阻塞项
- 选中系统应服务于对象/特征编辑，不应反过来驱动复杂视口系统扩张

### 4.3 UI 约束

- 继续复用当前 `test_viewer` 作为最小验证入口
- 当前版本的 UI 目标是验证结构与语义，不追求完整编辑器形态
- 若中途发现需要更复杂的对象树、gizmo 或屏幕拾取，不得直接插入本版本

## 5. 分步骤执行计划

### 步骤 1：建立正式的参数化对象与特征链结构

目标：

- 将当前“一个对象 = 一个基础 primitive + 固定 operators”的结构，升级为正式的对象级建模结构

建议范围：

- 为参数化对象增加稳定 id，例如：
  - `ParametricObjectId`
- 为特征增加稳定 id，例如：
  - `ParametricFeatureId`
- 明确对象级元数据：
  - `name`
  - `object kind`
  - `local transform`
  - `pivot/origin`
- 明确特征级元数据：
  - `feature kind`
  - `enabled`
  - `feature name`
  - 顺序语义
- 将当前 primitive 与 operator 统一收口到“特征链”表达中

完成判定：

- 参数化对象不再只是临时拼装的测试对象状态
- 每个对象和特征都具备稳定可引用 id
- 当前求值入口仍能从对象级结构稳定生成最终 mesh

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 2

### 步骤 2：在测试界面中接入对象列表与特征列表

目标：

- 让测试界面不再只展示固定对象卡片，而是开始显式表达对象层和特征层

建议范围：

- 在当前 `Scene` 页中加入最小对象列表
- 增加当前对象的特征列表
- 将当前对象的基础生成特征与修改特征按列表展示
- 提供最小 feature 操作入口：
  - 添加
  - 删除
  - 启用 / 禁用
- 保持现有 primitive / operator 参数入口可继续驱动当前对象

完成判定：

- UI 中可以显式看到“对象”与“特征”的层级
- 当前对象的特征列表可被观察与操作
- 对特征的增删与启停会进入统一求值链路

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 3

### 步骤 3：建立最小 `SelectionState` 与激活闭环

目标：

- 在不做屏幕拾取的前提下，把对象/特征的选中和激活语义建稳

建议范围：

- 引入 `SelectionState`
- 显式维护：
  - `selectedObjectId`
  - `activeObjectId`
  - `selectedFeatureId`
  - `activeFeatureId`
- 通过对象列表 / 特征列表驱动最小选中行为
- 让当前激活对象与现有聚焦 / 相机状态开始建立最小对应关系
- 明确对象删除、特征删除后的选中状态回退策略

完成判定：

- 当前对象与当前特征都可以被显式激活
- 删除对象 / 特征后，选中状态不会进入悬空状态
- 当前激活对象可作为后续 `Focus Selected`、控制点编辑和局部坐标的稳定宿主

测试与构建：

- 由用户执行
- 用户确认通过后本版本开发完成

## 6. 风险与注意事项

- 如果对象 id 与特征 id 设计不稳，后续控制点与拾取系统会持续返工
- 如果把“特征链结构化”做成一次性的大重构，容易影响当前稳定求值链路
- 如果在本版本中顺手引入复杂屏幕拾取或 gizmo，会显著扩大版本范围
- 如果删除对象 / 特征时不先定义选中回退规则，UI 状态很容易悬空

## 7. 版本完成后的收尾要求

- 若执行过程中形成新的明确讨论结论，需同步归档到 `TalkDoc/`
