# 0.0.14 开发计划

## 1. 版本目标

围绕当前测试界面的控制面板承载压力与相机 UI 规则分散问题，继续收口 `test_viewer` 的 UI 结构，使当前界面在较小窗口尺寸下仍具备稳定可操作性，同时把 `Projection + Zoom Mode + Active Parameter` 的规则从控件实现中进一步抽离，为后续 `Viewport Zoom` 接入提供更清晰的结构边界。

## 2. 本版本范围

### 2.1 包含内容

- 为 `ViewerControlPanel` 增加滚动承载能力
- 调整主窗口默认尺寸与控制面板承载策略
- 将相机模式相关 UI 规则整理为独立 helper / policy 落点
- 继续收口 `ViewerWindow` 中的 UI 同步与编排职责

### 2.2 不包含内容

- `Viewport Zoom`
- 模型参数变化后的：
  - 保持当前视角
  - 自动适配视图
  策略实现
- `Orthographic + Dolly` 的 UI 开放
- `render_core` / `render_gl` 协议层扩展
- 新的拾取、选中与调试绘制功能

## 3. 输入来源

本版本计划吸收以下评审输入：

- `ReviewDoc/0.0.13-评审遗留项.md`
- `TalkDoc/2026-03-12-控制面板滚动与窗口高度策略-讨论.md`
- `TalkDoc/2026-03-12-ZoomMode完整UI策略-讨论.md`
- `TalkDoc/2026-03-12-UI层代码结构是否继续拆解-讨论.md`

本版本明确不吸收以下后续输入：

- `Viewport Zoom`
- 模型参数变化策略
- 旋转中心与选中系统扩展

## 4. 已确认的设计约束

### 4.1 控制面板承载约束

- 当前右侧控制面板必须支持在较小主窗口高度下完整访问全部内容
- 优先通过滚动承载解决空间不足，不打散现有子面板结构
- 不在本版本引入新的复杂面板导航方式

### 4.2 相机 UI 规则约束

- `Projection`、`Zoom Mode`、`Active Parameter` 的规则表达不应继续散落在控件细节中
- `Distance`、`Vertical FOV`、`Ortho Height` 的主次角色语义必须保持一致
- 不允许因规则抽离导致当前 `Perspective + Dolly`、`Perspective + Lens`、`Orthographic + Lens` 行为回归

### 4.3 结构收口约束

- 不把新的相机交互语义直接插入当前版本
- 不把 `Viewport Zoom` 的状态模型提前混入当前 UI 重构
- `ViewerWindow` 继续只承担窗口编排与状态同步，不继续堆积规则判断

## 5. 分步骤执行计划

### 步骤 1：补齐控制面板滚动承载与窗口布局收口

目标：

- 为右侧控制面板增加滚动承载能力
- 调整主窗口默认尺寸或布局细节，使当前面板在常见窗口高度下可完整访问

完成判定：

- 控制面板内容在较小窗口下可通过滚动完整访问
- 当前对象、灯光、相机面板结构不回归
- 工程可正常编译

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 2

### 步骤 2：抽离相机模式 UI 规则层

目标：

- 将 `Projection + Zoom Mode + Active Parameter` 的规则从 `CameraControlWidget` 细节实现中进一步抽离
- 为后续继续扩展相机 UI 规则提供稳定落点

完成判定：

- 相机 UI 规则已有独立 helper / policy 或等价结构承载
- 当前三组已支持模式的状态回填与启停逻辑不回归
- 工程可正常编译

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 3

### 步骤 3：继续收口窗口层编排职责并整理后续输入

目标：

- 进一步减少 `ViewerWindow` 对控件细节与规则细节的直接依赖
- 将本版本收口结果整理为后续 `Viewport Zoom` 与模型变化策略的稳定输入

完成判定：

- `ViewerWindow` 的 UI 编排职责较当前更集中清晰
- 本版本开发记录可清晰说明承载策略与规则层调整
- 后续版本具备继续评审 `Viewport Zoom` 的清晰边界

测试与构建：

- 由用户执行
- 用户确认通过后本版本开发完成
