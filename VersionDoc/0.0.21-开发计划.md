# 0.0.21-开发计划
## 1. 版本目标

围绕 `0.0.20` 已经建立的参数化对象、特征链与最小选中闭环，继续推进到“正式对象增删与动态对象列表”的第一版。

本版本的目标是：

- 将当前固定 3 个测试对象的承载方式，推进为动态对象集合
- 为测试界面补齐最小对象增删入口
- 让对象列表真正成为对象层的主入口，而不是固定 3 个对象的只读映射
- 保持现有 feature 列表与最小 selected / active 语义继续可用

本版本完成后，项目应至少具备：

- 动态对象集合而非固定三对象数组
- 对象列表驱动的对象增删闭环
- 新增对象后的默认 primitive 初始化
- 删除对象后的 selection / active 回退闭环

## 2. 本版本范围
### 2.1 包含内容

- 将 `Viewport` 内部对象承载从固定数量结构升级为动态集合
- 为参数化对象建立最小默认创建入口：
  - `Add Box`
  - `Add Cylinder`
  - `Add Sphere`
- 支持删除当前选中对象
- 更新 `Scene` 页对象列表，使其承载动态对象数量
- 保持对象列表、feature 列表、selected / active 状态与对象增删联动
- 保持当前 mesh rebuild、bounds update、repository / GPU update、focus 逻辑在动态对象集合下继续成立

### 2.2 不包含内容

- 视口屏幕拾取
- 控制点显示与拖拽
- gizmo
- feature 顺序调整
- feature 重命名
- 完整对象树
- raw mesh component 编辑

## 3. 输入来源

本版本计划吸收以下输入：

- `VersionDoc/0.0.20-开发记录.md`
- `VersionDoc/0.0.20-开发计划.md`
- `TalkDoc/2026-04-03-参数化建模单元分类与选中编辑方案-讨论.md`
- `TalkDoc/2026-04-03-参数化建模与选中系统分期方案-讨论.md`

本版本明确吸收的结论包括：

- 当前下一优先级应转向正式对象增删，而不是提前进入控制点拖拽
- 对象层必须继续稳定在显式 id 与选中状态之上
- `selected` 与 `active` 状态在对象增删后必须有明确回退策略
- 测试界面应继续服务于语义验证，而不是直接演化成完整编辑器

当前未发现与本版本目标直接相关、且必须吸收的 `ReviewDoc/` 遗留项；如开发过程中发现新的非阻塞结构缺口，应按流程登记到 `ReviewDoc/`

## 4. 已确认的设计约束

### 4.1 动态对象集合约束

- 不能再以 `kSceneObjectCount = 3` 作为对象层的核心前提
- 对象集合切换为动态后，仍需保留稳定的 `ParametricObjectId`
- 新增对象必须通过统一对象创建入口建立默认 descriptor、默认 material 与默认 transform
- 删除对象后，必须同步处理：
  - repository item
  - mesh / material / texture 资源引用
  - selection state
  - feature list状态

### 4.2 UI 约束

- `Scene` 页继续复用当前 `Object Explorer`
- 新增对象入口优先放在对象列表附近
- 当前对象参数编辑区在本版本内可以继续沿用测试性质的承载方式，但必须保证不依赖固定三对象假设
- 如中途发现需要重做完整对象详情面板，不得直接扩进本版本

### 4.3 范围控制约束

- 本版本只解决对象层动态化与对象增删闭环
- 不得顺手扩展到控制点、拾取、高亮或 gizmo
- 若发现 feature 列表需要顺序调整等增强，登记为后续版本议题，不直接插入本版

## 5. 分步骤执行计划
### 步骤 1：将对象承载从固定三对象重构为动态对象集合
目标：

- 移除 `Viewport` 与 `PanelState` 对固定三对象数组的核心依赖，为对象增删建立数据结构基础

建议范围：

- 将对象承载改为动态集合
- 保留稳定对象 id 与现有参数化对象 descriptor
- 改造 bounds、focus、frame packet、selection state 等遍历逻辑，使其适配动态对象集合
- 保留当前默认场景仍以 3 个对象启动，但不再依赖固定数量常量驱动逻辑

完成判定：

- 对象集合不再以固定 3 个对象数组作为核心模型
- 默认场景仍可正确初始化为 3 个对象
- mesh / bounds / selection 基础逻辑在动态对象集合下仍成立

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 2

### 步骤 2：在测试界面接入对象增删入口并联动动态对象列表
目标：

- 让对象列表成为正式对象主入口，并支持最小的对象添加与删除操作

建议范围：

- 在 `Object Explorer` 中新增：
  - `Add Box`
  - `Add Cylinder`
  - `Add Sphere`
  - `Delete Selected Object`
- 新增对象后自动进入 selected 状态
- 删除对象后按回退规则更新 selected / active 对象
- 保持 feature 列表与对象详情区同步刷新

完成判定：

- UI 中可新增不同 primitive 类型对象
- UI 中可删除当前选中对象
- 对象列表、feature 列表和当前对象详情区能正确随增删同步刷新

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 3

### 步骤 3：收口对象增删后的状态闭环与默认行为
目标：

- 让对象增删与现有 `selected / active / focus / reset` 行为形成稳定闭环

建议范围：

- 明确删除对象后的 selection / active 回退策略
- 明确新增对象后的默认 transform / color / visibility 行为
- 确保 `Reset Defaults` 后回到稳定默认对象集
- 确保 `Focus Selected Object` 与 `Focus All` 在对象增删后行为仍稳定

完成判定：

- 对象增删后不会出现 selection / active 悬空
- `Reset Defaults` 后可稳定回到默认场景
- focus 相关行为在动态对象集合下仍保持正确

测试与构建：

- 由用户执行
- 用户确认通过后本版本开发完成

## 6. 风险与注意事项

- 如果动态对象集合改造不彻底，固定三对象假设会在 `ViewerWindow`、`ViewerControlPanel` 或 bounds/focus 逻辑中残留
- 如果对象删除时没有完整处理 repository item 与渲染资源，容易留下悬空句柄
- 如果对象参数编辑区继续绑定固定对象序号，会在对象新增后立刻出现错误映射
- 如果 selection 回退规则不先明确，对象删除后 UI 和状态很容易进入不一致
