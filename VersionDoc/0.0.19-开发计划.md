# 0.0.19-开发计划

## 1. 版本目标

围绕当前 `test_viewer` 测试/展示界面的信息架构问题，重构右侧控制面板的组织方式，使界面从“单一线性滚动面板”升级为“按分类组织的分组界面”，并将对象参数入口迁移到更符合对象编辑语义的位置。

本版本完成后，测试界面应具备：

- 顶层分类明确的控制面板结构
- 对象编辑入口与对象参数入口归位
- 相机操作、调试信息与对象编辑不再混排
- 现有 `Viewport` 联动、状态回填与核心交互行为不回归

## 2. 本版本范围

### 2.1 包含内容

- 为 `ViewerControlPanel` 引入顶层分类承载结构
- 将当前线性堆叠的面板重新组织为：
  - `Scene`
  - `Camera`
  - `Debug`
- 将 Box / Cylinder / Sphere 的 primitive 参数从全局 `Actions` 区迁移到对应对象面板
- 重组 `Bounds`、重置和聚焦类动作的承载位置，使其更符合语义分类
- 保持现有外部信号/槽联动尽量稳定，避免对 `Viewport` 行为链路造成非必要冲击

### 2.2 不包含内容

- 引入对象选择系统
- 引入节点式建模或历史栈承载
- 引入完整的 `ViewerControlBinder` / `ViewerUiStateAdapter`
- 扩展新的相机交互语义
- 重写现有 `Viewport Zoom`、模型变化视图策略或渲染链路

## 4. 已确认的设计约束

### 4.1 UI 结构约束

- 本版本优先重构信息架构，不追求一次性重写全部子控件
- `ViewerControlPanel` 继续负责总装
- 顶层分类应服务于“使用任务”而不是“代码文件边界”
- 右侧控制面板仍保持固定宽度和滚动承载能力

### 4.2 兼容性约束

- `ViewerWindow` 与 `ViewerControlPanel` 之间的既有信号/槽接口尽量保持稳定
- `Viewport -> Panel` 的状态回填链路不能因 UI 重组而失效
- 现有对象、相机、灯光、聚焦与重置能力不得回归

### 4.3 范围控制约束

- 本版本不把“界面重组”扩展成“大规模架构重写”
- 如果执行过程中发现 binder / adapter 层是优化点但非当前阻塞，必须登记到 `ReviewDoc/`
- 当前版本只处理与分组、分类直接相关的 UI 组织问题

## 5. 分步骤执行计划

### 步骤 1：重构 `ViewerControlPanel` 的顶层分类结构

目标：

- 将当前单列线性堆叠的内容改为带顶层分类的面板组织方式

建议范围：

- 为 `ViewerControlPanel` 引入顶层分类容器
- 形成至少三个分类页：
  - `Scene`
  - `Camera`
  - `Debug`
- 将现有对象面板与灯光面板归入 `Scene`
- 将现有相机面板归入 `Camera`
- 将 `Bounds` 与调试型入口迁入 `Debug` 或更明确的调试区

完成判定：

- 右侧控制面板不再是所有组件线性堆叠在一个内容流中
- 用户能够通过分类页快速定位“对象/场景”、“相机”、“调试”三类入口
- 现有面板状态回填和主要交互行为保持正常

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 2

### 步骤 2：将 primitive 参数入口迁移到对应对象分组

目标：

- 让对象的建模参数回到对象面板内，消除当前 `Actions` 区的职责混杂问题

建议范围：

- 扩展 `SceneObjectControlWidget`
- 将 Box 参数归入 Box 面板
- 将 Cylinder 参数归入 Cylinder 面板
- 将 Sphere 参数归入 Sphere 面板
- 保持现有 primitive 参数联动逻辑不变，只调整承载位置与局部组织方式

完成判定：

- `Actions` 不再承担三类 primitive 的参数编辑职责
- 用户编辑对象时，不需要跳转到全局动作区才能调整该对象的基础参数
- primitive 参数变更后的 mesh / bounds / 视图策略联动不回归

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 3

### 步骤 3：收口动作区与调试区的职责边界

目标：

- 清理剩余的全局动作入口，使界面分类在语义上收口

建议范围：

- 重新组织 `Focus All`、`Focus Sphere`、`Reset Defaults`、`Model Change View`
- 将相机相关动作归入 `Camera`
- 将只读信息与重置/调试型动作收口到 `Debug`
- 删除或弱化当前泛化的 `Actions` 命名与承载方式

完成判定：

- 界面中不再存在明显的“杂项动作堆放区”
- 相机动作、对象参数、调试信息的边界清晰
- 当前版本的 UI 重组目标形成闭环

测试与构建：

- 由用户执行
- 用户确认通过后本版本开发完成

## 6. 风险与注意事项

- 如果顶层分类容器选型不当，可能导致现有滚动体验退化
- 如果 primitive 参数迁移时直接修改过多外部接口，容易放大回归范围
- 如果在执行过程中顺手引入 binder / adapter 层，容易超出本版本的冻结计划
- 若发现非阻塞型结构优化点，按 `ReviewDoc/` 流程登记，不直接插入当前版本
