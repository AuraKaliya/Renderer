# 0.0.16 开发计划

## 1. 版本目标

围绕“模型参数变化后的视角保持与自动适配策略”继续扩展 `test_viewer` 的视口行为，并将当前 `localBounds / worldBounds / sceneBounds / clip` 的基础链路进一步收口，使模型变化后的相机处理不再混同为自动 `Focus All`，而是形成：

- 保持当前视角
- 自动适配视图

两种明确策略的最小闭环。

## 2. 本版本范围

### 2.1 包含内容

- 明确模型参数变化与 `Focus / Viewport Zoom` 的行为边界
- 为模型变化后的相机处理建立独立策略入口
- 第一版支持两种策略：
  - 保持当前视角
  - 自动适配视图
- 在模型变化后自动更新：
  - `localBounds`
  - `worldBounds`
  - `sceneBounds`
  - `near/far`
- 在测试界面中提供最小验证入口

### 2.2 不包含内容

- 完整拾取系统
- 旋转中心 / 选中系统扩展
- `Viewport Zoom` 的更高精度锚点升级
- `Orthographic + Dolly` 的 UI 开放
- `render_core` 可见集 / 裁剪前端扩展
- 更完整的 `virtual target / tumble pivot / workspace` 系统

## 4. 已确认的设计约束

### 4.1 模型变化策略约束

- 模型参数变化后的相机处理，不得直接等同于自动 `Focus All`
- 默认必须显式区分：
  - 保持当前视角
  - 自动适配视图
- “保持当前视角”语义下，默认保持不变的相机量包括：
  - 当前观察方向
  - 当前投影模式
  - 当前 zoom 路径语义
- 模型变化后即使保持当前视角，也应自动更新 bounds 与裁剪相关数据

### 4.2 Bounds 与裁剪约束

- `localBounds / worldBounds / sceneBounds` 必须保持链路清晰
- 模型变化后应先完成 bounds 更新，再决定是否触发自动 framing
- `near/far` 更新与自动 framing 应显式区分：
  - 更新裁剪范围
  - 自动适配视图

### 4.3 UI 入口约束

- 模型变化策略不应直接混入当前 `Zoom Mode` UI
- 第一版只提供最小验证入口，不在本版本扩大 UI 设计范围

## 5. 分步骤执行计划

### 步骤 1：建立模型变化策略的最小状态与调度入口

目标：

- 为模型变化后的相机处理建立独立于 `Focus` 与 `Viewport Zoom` 的最小策略入口
- 明确：
  - 保持当前视角
  - 自动适配视图
  的状态表达与调度边界

完成判定：

- 代码中已存在模型变化策略的最小状态或 helper 落点
- 当前 `Focus / Viewport Zoom / 普通缩放` 行为不回归
- 工程可正常编译

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 2

### 步骤 2：接入模型变化后的 bounds / clip 更新闭环

目标：

- 当模型参数发生变化时，自动完成 bounds 链与裁剪链更新
- 在“保持当前视角”语义下，不自动改写当前 framing

完成判定：

- 模型变化后可自动更新：
  - `localBounds`
  - `worldBounds`
  - `sceneBounds`
  - `near/far`
- 当前视角保持路径可独立验证
- 工程可正常编译

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 3

### 步骤 3：接入自动适配视图策略并完成版本收口

目标：

- 在模型变化后提供“自动适配视图”最小路径
- 明确与 `Focus All / Focus Point / Viewport Zoom` 的行为边界

完成判定：

- 模型变化后可选择自动适配视图
- 本版本开发记录可清晰说明：
  - 模型变化策略
  - `Focus`
  - `Viewport Zoom`
  三者边界
- 本版本最小闭环完成

测试与构建：
