# 0.0.18 开发计划

## 1. 版本目标

在 `0.0.17` 已建立的基础 primitive 参数对象化能力之上，引入最小建模算子栈第一版，使单个参数化对象可以由：

- 基础 primitive
- 少量可编辑算子

共同求值为最终 `MeshData`，并继续接入：

- bounds 更新
- GPU mesh 更新
- `keep_view / auto_frame`

形成“基础体 + 算子”的第一版非破坏性建模闭环。

## 2. 本版本范围

### 2.1 包含内容

- 建立对象级参数描述：
  - `ParametricObjectDescriptor`
  - `OperatorDescriptor`
- 建立最小算子协议与固定顺序求值入口
- 首批接入：
  - `Mirror`
  - `Linear Array`
- 在 `test_viewer` 中提供最小算子编辑入口
- 将算子变化接入：
  - mesh 重建
  - local bounds 更新
  - repository / GPU 更新
  - `keep_view / auto_frame`

### 2.2 不包含内容

- 完整 Modifier Stack 排序系统
- 完整 Construction History
- `Bevel`
- `Boolean`
- 曲线/剖面驱动建模
- 节点图系统
- 多对象依赖关系

## 3. 输入来源

本版本计划吸收以下输入：

- `VersionDoc/0.0.17-开发记录.md`
- `TalkDoc/2026-04-02-参数化建模路线参考Blender与Maya-讨论.md`
- `TalkDoc/2026-04-02-0.0.18最小建模算子栈范围-讨论.md`

本版本明确不吸收以下后续输入：

- 完整 stack 排序 UI
- `Bevel`
- `Boolean`
- 曲线/剖面驱动
- 实例化 / 程序化布置

## 4. 已确认的设计约束

### 4.1 算子系统约束

- 当前算子系统只服务于单对象求值
- 当前算子求值顺序固定，不做可拖拽排序
- `OperatorDescriptor` 必须与 `PrimitiveDescriptor` 分层
- 算子变化不允许绕开统一求值路径，必须进入：
  - `descriptor -> build -> mesh -> bounds -> GPU -> view strategy`

### 4.2 几何能力约束

- 当前版本优先选择参数简单、几何规则清晰的算子
- 首批算子应避免大规模拓扑修改
- 当前版本不引入复杂布尔或倒角算法

### 4.3 UI 约束

- 当前版本只提供最小 operator UI
- 不引入复杂 stack 面板、排序交互、批量对象编辑
- UI 目标是验证：
  - enable/disable
  - 最小参数修改
  - 结果联动

## 5. 分步骤执行计划

### 步骤 1：建立对象级 descriptor 与统一求值入口

目标：

- 将当前“primitive 直接求值”的结构升级为“基础 primitive + operators”对象级求值结构

建议范围：

- 新增：
  - `ParametricObjectDescriptor`
  - `OperatorKind`
  - `OperatorDescriptor`
  - `MirrorOperatorSpec`
  - `LinearArrayOperatorSpec`
- 为 `parametric_model` 增加对象级 `build(...)` 入口

完成判定：

- `parametric_model` 已可表达“一个基础体 + 多个算子”的对象描述
- 现有 `0.0.17` primitive 参数对象化路径不回归
- 当前项目仍保持 `parametric_model` 与渲染后端解耦

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 2

### 步骤 2：实现 Mirror / Linear Array 并接入重建闭环

目标：

- 让算子不只是协议存在，而是能真正修改求值结果

建议范围：

- 在 `parametric_model` 中实现：
  - `Mirror`
  - `Linear Array`
- 算子变化后自动完成：
  - 最终 mesh 重建
  - local bounds 更新
  - repository / GPU 更新
  - `keep_view / auto_frame` 联动

完成判定：

- `Mirror` 和 `Linear Array` 可真实改变几何结果
- 最终 bounds 能正确反映算子影响
- 视口中算子变化与视图策略联动正常

测试与构建：

- 由用户执行
- 用户确认通过后进入步骤 3

### 步骤 3：补最小 operator UI 并收口版本边界

目标：

- 在 `test_viewer` 中形成最小可验证的算子编辑界面
- 收口本版本只做“最小算子栈第一版”

建议范围：

Mirror：
- `Enabled`
- `Axis`

Linear Array：
- `Enabled`
- `Count`
- `Offset X`
- `Offset Y`
- `Offset Z`

完成判定：

- 用户可以在 `test_viewer` 中开关并调参首批算子
- 算子变化 -> 重建 -> bounds / 视图策略闭环完整
- 本版本开发记录可清晰说明：
  - 为什么只做 `Mirror + Linear Array`
  - 为什么 `Bevel / Boolean / Curve` 留待后续

测试与构建：

- 由用户执行
- 用户确认通过后本版本开发完成

## 6. 风险与注意事项

- 不允许在本版本中插入 `Bevel`
- 不允许在本版本中插入 `Boolean`
- 不允许在本版本中插入节点系统
- 若几何合并路径复杂度超出当前步范围，优先先打通最小正确结果
- 若出现当前步骤阻塞问题，按 `DebugDoc` 流程处理
- 若发现非阻塞优化点，按 `ReviewDoc` 流程登记

## 7. 版本完成后的收尾要求

当本版本全部步骤完成后，需要在 `VersionDoc/` 下生成：

- `0.0.18-开发记录.md`

并按 `ANGENT.md` 规则执行版本提交。
