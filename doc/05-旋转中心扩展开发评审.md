# 旋转中心扩展开发评审

## 1. 文档目的

本文档用于整理当前项目围绕“旋转中心”扩展开发所涉及的能力项、依赖关系、实施顺序与评审决策点。  
目标不是直接进入实现，而是先把后续开发边界、优先级和架构影响讲清楚，作为 `0.0.5` 及后续版本范围划分的依据。

## 2. 当前实现现状

当前项目已经具备以下基础能力：

- `OrbitCameraController` 已支持：
  - `rotate`
  - `zoom`
  - `pan`
- `orbitCenter` 已作为轨道相机的旋转中心参与计算
- `Viewport` 已支持：
  - 左键旋转
  - 滚轮缩放
  - 右键平移
- `Focus Sphere` 已能将观察中心切换到球体对象位置

当前仍存在以下限制：

- `orbitCenter` 仍是隐式状态，UI 中不可见
- 没有模型包围盒与场景总包围盒
- 没有屏幕拾取与选中系统
- 近远裁剪面没有基于目标范围自动更新
- 没有围绕旋转中心的调试可视化

## 3. 核心设计目标

围绕旋转中心扩展开发，建议后续能力建设满足以下目标：

1. 旋转中心有明确的数据定义和空间约定
2. 不同交互操作下，旋转中心更新逻辑一致且可预期
3. 相机、场景数据、包围盒、裁剪策略之间关系清晰
4. 不破坏当前模块独立性
5. 为后续“选中聚焦”“自动裁剪”“编辑器视口交互”留出扩展空间

## 4. 待开发项总览

### 4.1 相机与旋转中心能力

1. 旋转中心显式数据化
- 将当前隐式 `orbitCenter` 提升为清晰的状态接口
- 需要支持读取、设置、重置、聚焦到指定点

2. 旋转中心更新策略定义
- 明确平移、缩放、旋转、聚焦、选中等操作下中心是否变化
- 统一行为语义，避免后续功能引入时出现冲突

3. 相机状态补全
- 除 `distance` 与 `fov` 外，逐步增加：
  - `orbitCenter`
  - `camera position`
  - `yaw/pitch`
  - `near/far`

4. 旋转中心调试能力
- 提供可视化或可读状态，便于排查交互问题

### 4.2 空间与坐标体系

5. 明确空间定义
- 至少统一以下空间：
  - 模型空间
  - 世界空间
  - 相机空间
  - 裁剪空间

6. 明确旋转中心的空间约定
- 建议统一为世界空间坐标
- 避免在不同模块中出现不同空间语义

7. 补空间转换工具
- 为后续拾取、选中、调试绘制预留基础：
  - `world -> view`
  - `view -> clip`
  - `screen -> ray`

### 4.3 包围盒与场景范围数据

8. 增加局部包围盒协议
- 在共享层增加 `Aabb` 或等价结构
- 建模输出时同步提供局部 bounds

9. 增加世界包围盒计算
- 基于对象 transform + local bounds 生成 world bounds

10. 增加场景总包围盒
- 为 `Focus All`
- 为默认相机初始化
- 为自动裁剪提供依据

11. 增加包围球或等价范围描述
- 用于更稳定地估算聚焦距离和裁剪范围

### 4.4 选中与聚焦体系

12. 接入屏幕拾取
- 第一阶段建议优先做包围盒拾取
- 暂不必直接上三角级精确拾取

13. 增加选中状态管理
- 至少记录当前选中对象
- 后续扩展多选时再引入聚合策略

14. 支持以选中对象更新旋转中心
- `Focus Selected`
- 双击聚焦
- 设置中心为选中对象中心

15. 明确多选中心策略
- 后续如支持多选，需要定义中心计算规则

### 4.5 裁剪与投影更新

16. 基于目标范围自动计算 `near/far`
- 当前裁剪参数较静态
- 后续应与相机距离、目标范围联动

17. 聚焦时自动更新距离与裁剪参数
- `Focus Sphere`
- `Focus Selected`
- `Focus All`

18. 旋转过程中的裁剪稳定性
- 目标接近 near plane 时，需要避免被切掉

19. 缩放过程中的裁剪联动
- 缩放过近时应自动调整 near plane

20. 大尺度场景的深度精度预留
- 当前可先不做高级深度方案
- 但设计上应避免把裁剪参数完全写死

### 4.6 可见性与渲染前端扩展

21. 视锥裁剪
- 基于对象 bounds 做 frustum culling
- 建议放在 `render_core` 的装配阶段

22. 可见集构建扩展
- 当前主要依赖显隐标记
- 后续应支持 bounds 参与可见性判断

23. 旋转过程中的稳定剔除
- 避免因边界误差产生抖动与闪烁

### 4.7 UI 与调试工具

24. 面板中显示旋转中心
- 初期至少支持只读显示
- 后续可扩展为可编辑

25. 面板中显示更多相机状态
- `Yaw/Pitch`
- `Near/Far`
- `Camera Position`

26. 增加调试绘制
- 旋转中心点
- 选中对象包围盒
- 场景总包围盒

27. 增加交互提示
- 让当前视口交互模式对用户可见

### 4.8 架构层扩展

28. 共享层增加 Bounds 协议
- 建议在 `scene_contract` 中定义
- 保持建模、渲染前端、测试界面共享同一语义

29. `render_core` 增加范围数据支持
- `SceneRepository` 后续应能保存 bounds 等元数据

30. `FrameAssembler` 接入裁剪输入
- 为后续视锥裁剪和可见集构建做准备

31. `test_viewer` 继续从演示窗口向轻量视口演进
- 交互复杂度继续上升后，建议拆出独立交互层

## 5. 依赖关系梳理

推荐的依赖顺序如下：

1. 先补 bounds 协议与 mesh/local bounds
2. 再补世界包围盒与场景总包围盒
3. 再做聚焦逻辑：
   - `Focus All`
   - `Focus Selected`
   - `Focus Point`
4. 再做自动裁剪参数更新
5. 最后再接选中拾取与更复杂的可见性裁剪

原因：

- 没有 bounds，聚焦和裁剪很难做稳
- 没有统一空间语义，选中与中心切换会很容易失控
- 先把“数据基础”和“相机基础”补齐，再做交互，后续返工最少

## 6. 建议优先级

### 第一优先级

- `orbitCenter` 显式化
- bounds 协议
- mesh/local bounds
- world bounds
- scene bounds

### 第二优先级

- `Focus All`
- `Focus Selected`
- `Focus Point`
- 自动更新 `near/far`

### 第三优先级

- 包围盒拾取
- 选中状态管理
- 调试绘制
- 视锥裁剪

## 7. 建议分阶段实施

### 阶段 A：范围数据基础

目标：
- 在共享层补齐 bounds 协议
- 在建模输出阶段补 local bounds
- 在场景层补 world bounds 与 scene bounds

### 阶段 B：旋转中心显式化与聚焦能力

目标：
- 把 `orbitCenter` 从隐式状态提升为显式状态
- 支持：
  - `Focus All`
  - `Focus Selected`
  - `Focus Point`

### 阶段 C：自动裁剪与稳定视图

目标：
- 根据目标范围自动调整 `near/far`
- 提高旋转和缩放过程中的裁剪稳定性

### 阶段 D：选中与拾取

目标：
- 接入包围盒拾取
- 接入选中状态
- 接入“以选中对象为旋转中心”

### 阶段 E：调试与优化

目标：
- 增加旋转中心和 bounds 的调试绘制
- 逐步接入视锥裁剪

## 8. 架构约束建议

后续扩展中建议坚持以下约束：

1. `orbitCenter` 统一使用世界空间
2. `scene_contract` 中只定义数据协议，不出现 OpenGL 对象
3. `parametric_model` 输出 bounds 数据，但不依赖渲染后端
4. `render_core` 负责场景范围数据和裁剪输入的组织
5. `test_viewer` 只做交互与调试展示，不承载过多渲染前端逻辑

## 9. 需要评审拍板的事项

在进入 `0.0.5` 前，建议优先拍板以下问题：

1. `orbitCenter` 是否统一定义为世界空间坐标
2. 第一阶段是否只做 AABB，不引入更复杂 bounds
3. 第一阶段的聚焦中心是否统一取包围盒中心
4. 第一阶段拾取是否只做包围盒拾取
5. `near/far` 是否采用自动计算，同时保留手动覆盖入口
6. 是否需要在测试阶段就加入旋转中心和包围盒调试绘制

## 10. 对 0.0.5 的建议结论

如果按“最稳、返工最少”的路线推进，建议 `0.0.5` 不直接上拾取，而是先聚焦以下内容：

- bounds 协议与范围数据基础
- `orbitCenter` 显式化
- `Focus All / Focus Point`
- 基于范围的基础裁剪参数更新

这样可以先把旋转中心相关的底层模型做扎实，再进入选中与拾取。
