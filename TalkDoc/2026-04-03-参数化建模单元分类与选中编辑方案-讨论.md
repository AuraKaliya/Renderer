# 参数化建模单元分类与选中编辑方案讨论

## 1. 讨论背景

当前项目已经具备：

- 基础 primitive 参数对象化
- 最小建模算子栈
- `descriptor -> build -> mesh -> bounds -> view strategy` 的基础闭环

但仍缺少更正式的“对象级参数化建模编辑系统”，尤其缺少：

- 建模单元分层
- 选中系统
- 控制点与局部坐标
- 编辑 / 添加 / 删除的统一模型

用户进一步提出，后续参数化建模需要显式考虑：

1. 单元分类
2. 坐标控制点
3. 选中、编辑、添加、删除

并希望整体思路与经典建模软件靠拢，可参考 Blender、Maya 等软件。

## 2. 外部参考结论

### 2.1 Blender 的启发

Blender 官方手册明确区分：

- `Object Mode`
- `Edit Mode`
- 多对象编辑
- active object 与 selected objects

同时，Blender 的 `Modifier` / `Geometry Nodes` 路径说明了：

- 基础对象与后续建模操作应分层
- 操作顺序本身是建模语义的一部分
- 一个对象可以挂接多个可编辑修改器输入，而不是只保留最终 mesh

### 2.2 Maya 的启发

Maya 官方帮助强调：

- `Object` / `Component` 两级选择模式
- polygon / NURBS 的 component 选择
- construction history 可让已完成命令保持可编辑
- 历史节点的输入属性可被重新编辑，模型结果随之更新

这说明 Maya 的核心不是“直接改结果 mesh”，而是：

- 选择对象或组件
- 操作一个带历史的建模命令
- 通过输入节点继续编辑早期步骤

### 2.3 对当前项目的直接启发

当前项目不应直接跳到完整节点图编辑器，也不应先走纯 mesh 顶点编辑路线。

更适合的交集方案是：

1. 先把“参数化对象”定义清楚
2. 再把“对象内部的建模单元”定义清楚
3. 再把“控制点与局部坐标系”挂到这些单元上
4. 最后由选中系统驱动编辑、添加、删除

## 3. 建议采用的总体方案

建议将后续参数化建模系统建成三层编辑对象：

1. 对象层
2. 特征层
3. 控制点层

这是一个更接近 Blender `Object + Modifier` 与 Maya `Object + History Node + Component` 交集的方案。

### 3.1 对象层

对象层代表一个完整的参数化建模对象。

建议对象至少包含：

- `ObjectId`
- 对象名 / 类型
- 局部坐标系
- 基础生成单元
- 特征链
- 输出结果：
  - preview mesh
  - local bounds
  - world bounds

对象层承担的职责：

- 作为场景中的可选中对象
- 作为建模单元树的根
- 作为局部坐标系与 pivot 的主要宿主

### 3.2 特征层

特征层代表对象内部一个可编辑的建模步骤。

建议首批特征分类为：

- 生成类
  - Box
  - Cylinder
  - Sphere
  - Curve / Profile（后续）
- 修改类
  - Mirror
  - Linear Array
  - Radial Array（后续）
  - Bevel（后续）
- 辅助类
  - Reference Plane
  - Construction Axis
  - Pivot Override

每个特征应具备：

- `FeatureId`
- `FeatureKind`
- 输入参数
- 生效开关
- 局部坐标 / 参考坐标
- 控制点集合

特征层的意义在于：

- 它比纯 mesh component 更贴近参数化建模
- 它比单一 `PrimitiveDescriptor` 更适合承载“添加 / 删除 / 重排 / 编辑”

### 3.3 控制点层

控制点层不是“结果 mesh 顶点”，而是“驱动参数和局部形态的编辑点”。

建议控制点分成三类：

1. 尺寸控制点
- 例如 Box 的宽高深拉点
- Cylinder 的半径 / 高度拉点

2. 结构控制点
- 曲线的 CV / anchor / tangent
- profile 平面的定位点
- sweep 路径点

3. 参考控制点
- pivot
- 局部原点
- 操作平面中心
- 阵列方向点

每个控制点建议包含：

- `ControlPointId`
- 所属 `ObjectId`
- 所属 `FeatureId`
- `ControlPointRole`
- 位置
- 方向 / 法线 / 切线（按需）
- 约束类型
- 坐标空间

## 4. 单元分类建议

结合用户提出的“单元分类”诉求，建议当前项目采用以下分类体系。

### 4.1 运行时分类

1. `Object Unit`
- 场景中一个完整参数化对象

2. `Feature Unit`
- 对象内部一个可编辑建模步骤

3. `Control Unit`
- 驱动 feature 参数的控制点 / 控制柄

4. `Result Unit`
- 求值后的 mesh / bounds / preview 数据

其中，`Result Unit` 是输出，不建议作为第一阶段主编辑目标。

### 4.2 选择层级分类

建议将交互选择模式设计为：

1. `Object Mode`
- 选中整个对象
- 用于移动对象、复制对象、删除对象、切换当前编辑对象

2. `Feature Mode`
- 选中特征节点
- 用于编辑当前特征参数、启用/禁用、重排、删除

3. `Control Point Mode`
- 选中特征控制点
- 用于直接拖拽、局部变形、局部坐标操作

4. `Mesh Component Mode`
- 后续阶段再考虑
- 不建议在当前项目的下一阶段优先实现

这是一个推断方案：它不是 Blender 或 Maya 的逐字照搬，而是基于两者共同模式对当前项目做的收敛。

## 5. 坐标控制方案

### 5.1 必须引入局部坐标系

如果后续要支持控制点编辑与对象级建模，单纯依赖世界坐标会很快失控。

建议为对象和特征都显式提供局部坐标：

- `ObjectTransform`
- `FeatureLocalFrame`

这样可以支持：

- Mirror 平面偏移和方向
- Array 的局部方向
- 曲线 / 剖面类特征的局部工作平面
- 控制点的局部拖拽与 gizmo 对齐

### 5.2 控制点应带坐标空间标签

建议控制点显式声明它的编辑空间：

- `world`
- `object_local`
- `feature_local`

例如：

- Box 尺寸控制点通常属于 `object_local`
- 阵列方向点通常属于 `feature_local`
- 场景级辅助定位点可能属于 `world`

### 5.3 pivot / origin 不应继续隐式化

后续若要做选中和编辑，建议把以下状态显式建模：

- object origin
- feature pivot
- active gizmo frame

否则“选中谁、绕谁变换、控制点在哪个空间里移动”会不断产生语义冲突。

## 6. 编辑、添加、删除方案

### 6.1 编辑

编辑应分三种路径：

1. 参数编辑
- 改 feature 参数
- 例如 `radius`、`count`、`offset`

2. 控制点编辑
- 拖动控制点，反算参数
- 例如拖 Box 的 X 向拉点，得到新的 width

3. 坐标编辑
- 改对象 / 特征局部坐标
- 例如改 pivot、工作平面或特征方向

### 6.2 添加

建议添加操作只对“特征层”开放，而不是直接对结果 mesh 操作。

首批建议支持：

- 给对象添加一个生成特征
- 给对象添加一个修改特征
- 给曲线型特征添加控制点

例如：

- `Add Feature -> Mirror`
- `Add Feature -> Linear Array`
- `Add Control Point -> Curve CV`

### 6.3 删除

建议删除操作按层级区分：

1. 删除对象
- 删除整个参数化对象

2. 删除特征
- 从特征链中移除一个 feature

3. 删除控制点
- 仅在允许拓扑重建的 feature 中开放
- 例如曲线点、剖面点

不建议第一阶段开放“删除结果 mesh 顶点/边/面”，因为这会把系统拖向直接建模而不是参数化建模。

## 7. 选中系统方案

### 7.1 选中状态建议

建议统一维护：

- `SelectionMode`
- `selectedIds`
- `activeId`
- `hoveredId`

其中 `activeId` 很重要，建议对齐 Blender 的 active object / active element 思路。

### 7.2 可选中的目标类型

建议拾取系统后续支持以下目标：

- object
- feature
- control_point
- gizmo_handle

第一阶段不建议把最终 mesh component 作为主拾取目标。

### 7.3 选中行为建议

建议首批行为：

- 单选
- `Shift` 多选
- active item 显式高亮
- 删除当前选中对象或特征
- 以当前选中对象或特征更新 orbit / focus

后续再考虑：

- 框选
- 选中转换
- 软选择
- 约束选择

## 8. 与当前项目最匹配的实现路线

建议分三阶段推进。

### 8.1 第一阶段：对象级参数化编辑模型

目标：

- 先把“对象 / 特征 / 控制点”数据模型立起来

建议内容：

- `ParametricObject`
- `FeatureDescriptor`
- `ControlPointDescriptor`
- `SelectionState`
- `Object / Feature / ControlPoint` 三层 id 体系

这一步先不追求完整拾取和 gizmo，只先把编辑语义建好。

### 8.2 第二阶段：最小选中与编辑闭环

目标：

- 在测试界面中形成对象级和特征级编辑闭环

建议内容：

- 对象选中
- 特征选中
- active object / active feature
- 以选中对象聚焦
- feature 的添加 / 删除 / 编辑

### 8.3 第三阶段：控制点与局部 gizmo

目标：

- 让参数化建模从“面板改参数”升级为“视口中直接编辑”

建议内容：

- 控制点显示
- 控制点拾取
- 按局部坐标拖拽
- 控制点反推 feature 参数
- 基础工作平面 / pivot / axis 显式化

## 9. 关键结论

- 当前项目下一阶段不建议先做 raw mesh 顶点建模，而应先做“对象 + 特征 + 控制点”的参数化编辑模型
- 单元分类建议采用：
  - 对象单元
  - 特征单元
  - 控制单元
  - 结果单元
- 选择模式建议采用：
  - Object Mode
  - Feature Mode
  - Control Point Mode
  - Mesh Component Mode（后续）
- 控制点应直接服务于参数化 feature，而不是直接等同于最终 mesh 顶点
- 坐标系统必须显式化：
  - object local
  - feature local
  - world
- 经典软件对齐的重点，不是照搬完整 UI，而是吸收它们的三条核心原则：
  - 对象与组件分层
  - 可编辑历史/特征链
  - 基于局部坐标与控制点的直接编辑

## 10. 对后续版本规划的影响

如果采纳本讨论结论，后续版本更合适的方向不是继续补更多 primitive，而是优先进入：

1. 参数化对象的正式对象级描述
2. 特征链与控制点描述
3. 最小选中系统
4. 视口中的控制点编辑闭环

在这个基础上，再继续讨论：

- 曲线 / 剖面驱动建模
- 更正式的 gizmo 系统
- 更完整的 component / soft selection
