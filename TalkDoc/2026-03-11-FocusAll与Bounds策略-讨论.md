# Focus All 与 Bounds 策略讨论

## 1. 讨论背景

在准备推进 `Focus All / Focus Point` 接入测试界面时，需要先明确以下问题：

- `Focus All` 是否自动重算相机距离
- `Focus Point` 是否保持当前距离不变
- `near/far` 是否只在聚焦动作时更新
- `Focus Point` 是否提供世界坐标输入
- 包围球与 `Aabb` 的代码组织关系

这些结论将直接作为 `0.0.7` 开发计划的输入。

## 2. 讨论内容

本次讨论围绕以下核心点展开：

1. `Focus All` 的目标语义和相机摆位方式
2. `Focus Point` 的目标语义和距离策略
3. 聚焦动作与 `near/far` 更新的联动边界
4. `Focus Point` 在测试界面的最小 UI 形式
5. 包围球是否应与 `Aabb` 合并为同一具体类型

## 3. 讨论结论

### 3.1 聚焦策略

- `Focus All`：自动重算相机距离
- `Focus Point`：第一版保持当前距离不变
- `near/far`：第一版只在聚焦动作时更新，不做每帧自动更新
- `Focus Point`：第一版提供世界坐标输入

### 3.2 Bounds 组织策略

- 包围球与 `Aabb` 可以归为同一类“范围数据 / Bounds”代码
- 但不建议将两者合并为同一个具体数据结构
- `Aabb` 与 `BoundingSphere` 应保持各自独立的字段与运算语义
- 可以在同一命名空间、同一头文件组或同一工具模块下统一组织

### 3.3 当前阶段的实现策略

- 当前阶段不强制将 `BoundingSphere` 提升为共享协议层类型
- `0.0.7` 可先基于现有 `Aabb` 推导临时包围球半径，用于 `Focus All` 的距离估算
- 后续若选中、裁剪、剔除等场景明确需要复用，再正式引入 `BoundingSphere`

## 4. 对当前版本规划的影响

- `0.0.6` 无需回改
- `0.0.7` 可以围绕以下方向展开：
  - `Focus All` 接入测试界面
  - `Focus Point` 接入测试界面
  - 基于 `Aabb` 的拟合距离计算
  - 聚焦动作触发的基础 `near/far` 更新

## 5. 后续建议

建议 `0.0.7` 采用以下路线：

1. 在 `test_viewer` 中增加聚焦计算工具
2. 基于 `Aabb` 计算中心与拟合距离
3. 将 `Focus All` 接入 `Actions` 区域
4. 在相机面板中加入 `Focus Point` 的世界坐标输入
5. 让 `Focus Sphere` 改走统一聚焦链路
