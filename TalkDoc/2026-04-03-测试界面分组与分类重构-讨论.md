# 测试界面分组与分类重构讨论

## 1. 讨论背景

当前 `test_viewer` 的右侧控制面板虽然已经完成了组件化拆分与滚动承载，但整体仍然以线性堆叠方式组织：

- 三个对象面板顺序排列
- `Lighting`
- `Camera`
- `Bounds`
- `Actions`

随着参数化建模入口与相机/调试入口持续增长，当前界面已经出现新的使用问题：

- 所有入口都堆在同一条滚动流中，查找成本高
- primitive 参数仍集中在 `Actions` 区，和对象本身的编辑语义脱节
- 只读调试信息与高频编辑操作混排，界面层级不清晰

用户明确提出，当前测试/展示界面需要改为按分组、分类组织，而不是继续把所有操作组件放在一个面板中。

## 2. 讨论内容

### 2.1 当前结构问题

当前 `ViewerControlPanel` 的主要问题不是“没有子组件”，而是“总装层的信息架构仍是线性面板”。

具体表现为：

- `SceneObjectControlWidget` 已经承担对象可见性、旋转、颜色和算子编辑，但对象本身的 primitive 参数没有回到对象面板内
- `CameraControlWidget` 已经完成局部分组，但其相关的聚焦动作仍和其它动作混在 `Actions`
- `Bounds` 属于明显的只读调试信息，却和可编辑参数处于同一主滚动流的同等层级

### 2.2 可选方案比较

本次讨论主要比较了三类方案：

#### 方案 A：维持单列滚动，仅调整顺序

优点：

- 改动最小
- 基本不影响现有信号与状态结构

缺点：

- 只能缓解，不能解决“信息架构平铺”的核心问题
- 后续继续增加入口时仍会回到同样的问题

结论：

- 不建议作为本次版本主方案

#### 方案 B：继续保留单列，但大量引入折叠分组

优点：

- 可以一定程度压缩视觉高度
- 仍可复用现有面板布局

缺点：

- 结构上仍是单一滚动流
- 用户仍然需要在同一列中寻找“对象编辑 / 相机 / 调试”的不同语义区域
- 折叠状态管理会增加额外复杂度

结论：

- 可作为局部补充，但不适合作为主组织方式

#### 方案 C：引入顶层分类页，并在分类页内保持局部分组

优点：

- 可以先按使用任务分流，再在页内做局部结构
- 更适合当前测试界面逐步增长的功能形态
- 不要求一次性重写所有子控件

缺点：

- `ViewerControlPanel` 的总装结构需要调整
- primitive 参数入口需要迁移

结论：

- 适合作为当前版本主方案

## 3. 讨论结论

本次讨论确定采用“顶层分类 + 分类内局部分组”的重构方式。

### 3.1 顶层分类

右侧控制面板改为三个顶层分类页：

1. `Scene`
- 承载对象编辑与灯光编辑

2. `Camera`
- 承载投影模式、缩放模式、聚焦行为和相机状态

3. `Debug`
- 承载 bounds、重置与其它只读/调试型入口

### 3.2 对象参数归位

将当前位于 `Actions` 区中的 primitive 参数入口迁回对象面板：

- Box 参数归入 Box 面板
- Cylinder 参数归入 Cylinder 面板
- Sphere 参数归入 Sphere 面板

这样每个对象面板都围绕“这个对象如何构成与变化”展开，而不是把对象建模参数散落在全局动作区。

### 3.3 保持现有外部接口尽量稳定

本次版本优先调整 UI 组织方式，不以大规模重写信号协议为目标。

约束如下：

- `ViewerControlPanel` 继续作为总装层
- 现有 `ViewerWindow -> ViewerControlPanel` 连接关系尽量保持可复用
- `PanelState` 可以继续保留现有整体结构，只在内部按分类页回填

### 3.4 本次版本明确不做的内容

为防止范围扩张，本次讨论明确以下内容不纳入当前版本：

- 引入正式的对象选择系统
- 引入节点式建模 UI
- 引入完整的 `ViewerControlBinder` / `ViewerUiStateAdapter`
- 对 `Viewport` 交互语义做新的扩展
- 对 `camera_mode_ui_policy` 做额外功能性扩张

## 4. 对当前版本的影响

该讨论结论将直接作为 `0.0.19` 的输入来源，用于规划测试界面的分组与分类重构版本。

预期本次版本应至少完成：

- `ViewerControlPanel` 从线性面板改为顶层分类承载
- primitive 参数从 `Actions` 迁移到对象分组
- `Bounds` / 重置等调试入口从高频编辑区剥离

## 5. 后续建议

如果本次版本完成后界面组织已经稳定，后续版本再继续评估：

- 是否引入更正式的 `ScenePanelWidget / CameraPanelWidget / DebugPanelWidget`
- 是否把 `ViewerWindow` 中的信号绑定进一步抽离为 binder / adapter 层
- 是否为对象参数面板继续增加更明确的“基础参数 / 外观 / 算子”内部分区
