# 2026-04-02-ViewportZoom采用Blender式工作窗口缩放-讨论

## 1. 讨论背景

在 `0.0.15` 版本规划中，原先将 `Viewport Zoom` 倾向定义为：

- 独立于 `Dolly` / `Lens` 的概念
- 偏屏幕空间的视口放大
- 不直接改变相机状态

但在继续讨论后，已明确当前项目更希望得到的是：

- 一个缩放后的局部工作窗口
- 该窗口在缩放后仍能继续进行：
  - 旋转
  - 平移
  - 后续缩放
- `Viewport Zoom` 会影响相机工作状态，而不是单纯视觉放大

因此需要重新定义 `Viewport Zoom` 的语义与实现边界。

## 2. 当前倾向

当前倾向采用更接近 Blender 一类 DCC 软件的做法。

参考方向包括：

- 鼠标位置作为缩放锚点
- 缩放时联动更新当前工作中心
- 缩放后的视口仍然是可继续工作的窗口
- 后续 orbit / pan / zoom 基于新的工作中心继续成立

这类行为更接近：

- `Zoom to Mouse`
- `Auto Depth`
- 基于兴趣点 / 工作中心继续导航

而不是纯粹的屏幕空间放大。

## 3. 新的 Viewport Zoom 语义

### 3.1 概念定位

`Viewport Zoom` 仍然是独立于：

- `Dolly`
- `Lens`

的第三类缩放语义。

但它不再被定义为“纯屏幕空间放大”，而是：

- 一种相机工作窗口缩放
- 一种围绕鼠标锚点进行的局部工作区建立方式

### 3.2 行为结果

执行 `Viewport Zoom` 后：

1. 当前关注区域被收口为新的局部工作窗口
2. 该窗口仍能继续承接：
   - orbit
   - pan
   - zoom
3. 后续相机交互的工作中心应与本次缩放结果保持连续

### 3.3 对相机状态的影响

与此前“尽量不改相机”的方向不同，当前结论是：

- `Viewport Zoom` 会改变相机相关状态
- 尤其会影响工作中心，当前优先考虑通过 `orbitCenter` 体现
- 但它不等同于普通 `Dolly`
- 也不等同于普通 `Lens`

## 4. 推荐的第一版实现思路

### 4.1 实现原则

第一版不追求完整独立的视口空间系统，优先采用：

- 鼠标锚点驱动
- `orbitCenter` 补偿
- 复用现有 `Dolly` / `Lens` 部分路径

也就是说：

- 在语义上独立
- 在实现上允许作为特殊交互流程存在

### 4.2 建议的最小实现路径

建议的第一版路径为：

1. 根据鼠标位置确定当前工作锚点
2. 调整 `orbitCenter`，使工作中心向该锚点收拢
3. 再执行当前选定缩放路径中的实际缩放
   - 可复用 `Dolly`
   - 或复用 `Lens`
4. 缩放结束后，新的工作中心继续承接后续 orbit / pan / zoom

### 4.3 为什么不直接做纯屏幕放大

原因：

- 纯屏幕放大不能自然承接后续 orbit / pan
- 无法很好解释“缩放后的窗口仍然是可工作的窗口”
- 会把 `Viewport Zoom` 做成视觉效果，而不是导航能力

## 5. 鼠标锚点的第一版建议

当前不建议第一版直接依赖完整拾取系统。

建议优先采用：

- 鼠标射线与经过当前 `orbitCenter` 的工作平面交点

原因：

- 能提供相对稳定的锚点
- 不需要先引入完整对象拾取
- 能较自然地与当前轨道相机结构兼容

后续如果需要更高精度，再考虑：

- 鼠标下对象交点
- 可见表面深度
- 更完整的工作平面 / 工作坐标系系统

## 6. 与当前系统的边界

### 6.1 与 Dolly 的边界

- `Dolly` 是普通相机距离变化
- `Viewport Zoom` 可以内部复用 `Dolly`，但会额外带有鼠标锚点与工作中心重建语义

### 6.2 与 Lens 的边界

- `Lens` 是普通投影视体变化
- `Viewport Zoom` 可以内部复用 `Lens`，但不会被定义为镜头参数本身

### 6.3 与 Focus 的边界

- `Focus Point / Focus Bounds / Focus All` 是 framing 语义
- `Viewport Zoom` 是局部工作窗口建立语义
- 两者都可能改变工作中心，但意图不同

## 7. 对 0.0.15 的影响

`0.0.15` 的既有计划需要调整：

1. 不再使用“纯屏幕空间视口放大”的定义
2. 不再强调“Viewport Zoom 不改变相机”
3. 改为：
   - `Viewport Zoom` 会影响相机工作中心
   - 第一版允许复用现有 `Dolly` / `Lens` 路径
   - 第一版重点验证缩放后窗口的连续可操作性

## 8. 讨论结论

- 当前项目的 `Viewport Zoom` 倾向采用 Blender 式工作窗口缩放路线
- `Viewport Zoom` 是独立概念，但不是纯视觉放大
- 它会影响相机工作状态，尤其是工作中心
- 第一版可作为“鼠标锚点 + orbitCenter 补偿 + 复用现有缩放路径”的特殊实现
- 第一版锚点建议优先采用“鼠标射线与 orbitCenter 工作平面交点”
