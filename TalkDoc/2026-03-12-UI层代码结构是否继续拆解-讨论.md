# UI 层代码结构是否继续拆解讨论

## 1. 讨论背景

当前 `test_viewer` 的 UI 层已经完成过一轮拆分，主要结构包括：

- `ViewerWindow`
- `ViewerControlPanel`
- `SceneObjectControlWidget`
- `LightingControlWidget`
- `CameraControlWidget`

同时，相机与聚焦相关逻辑已经逐步从界面层抽离到：

- `Camera`
- `OrbitCameraController`
- `camera_focus_utils`
- `camera_clip_utils`
- `camera_focus_bounds_utils`

因此，本次讨论的核心不是“是否仍是单体 UI”，而是：

- 当前拆分是否已经足够
- 后续是否需要继续拆层
- 继续拆时，应该拆到什么粒度

## 2. 当前结构判断

### 2.1 当前 UI 层已经不是单体结构

从代码边界看，当前 UI 层已经具备基础组件化能力：

- `ViewerWindow` 负责窗口装配与 `Viewport`/面板连接
- `ViewerControlPanel` 负责面板总装
- 三类子控件分别承接对象、光照、相机控制

这说明“第一轮拆解”已经完成，当前问题不再是“是否需要从 0 开始拆”，而是“是否要进行第二轮职责收紧”。

### 2.2 当前的主要压力点集中在相机面板

相比对象面板和光照面板，当前最容易继续膨胀的是 `CameraControlWidget`，因为它已经开始同时承载：

- 投影模式显示与切换
- 缩放模式显示与切换
- `Distance`
- `Vertical FOV`
- `Ortho Height`
- `Near/Far`
- `Orbit Center`
- `Focus Point` 输入

后续如果继续加入：

- `Viewport Zoom`
- 视角保持 / 自动适配策略
- 正交模式细化规则

那么相机面板会先成为新的复杂度热点。

### 2.3 当前 `ViewerWindow` 仍然承担了较多“连接与回填编排”

虽然渲染和计算逻辑已经往 helper/控制器侧转移，但 `ViewerWindow` 仍然负责：

- UI 信号连接
- `Viewport` 状态回填
- 面板同步时机
- 各类 focus/zoom 结果刷新

这部分当前仍可接受，但已经接近“第二轮拆层”的触发点。

## 3. 是否需要继续拆解

结论：

- 需要继续拆解
- 但不应做“大范围 UI 重构”
- 应做“按职责继续收紧边界”的小步拆解

也就是说，当前不适合把整个 UI 层重新推翻重做；更合理的是：

1. 保留现有组件化布局
2. 将最容易膨胀的职责进一步抽离
3. 让 `ViewerWindow` 和 `CameraControlWidget` 继续减负

## 4. 不建议做的拆法

### 4.1 不建议把每个单独参数都拆成独立 widget

例如：

- `DistanceWidget`
- `FovWidget`
- `NearFarWidget`
- `OrbitCenterWidget`

这种拆法粒度太细，会带来：

- 信号数量膨胀
- 组装复杂度上升
- 阅读成本增加

当前项目体量还不需要做到这种颗粒度。

### 4.2 不建议把 `Viewport` 事件逻辑重新塞回 UI 层

相机交互、clip/focus 逻辑已经分离出来，这条方向是对的。

如果为了“UI 统一管理”再把它们往面板或窗口层回塞，会破坏之前已经建立好的边界。

## 5. 建议继续拆的部分

### 5.1 将 `CameraControlWidget` 继续按“状态展示区”和“行为控制区”分层

当前最值得继续拆的不是整个 `ViewerControlPanel`，而是 `CameraControlWidget`。

建议后续可拆成两块内部区域：

1. `CameraModeSection`
- `Projection`
- `Zoom Mode`
- 活跃参数规则

2. `CameraFocusSection`
- `Near/Far`
- `Orbit Center`
- `Focus Point`

这样做的好处是：

- `Zoom Mode UI` 的演进不会和 `Focus` 区混在一起
- 后续加入 `Viewport Zoom` 时，也更容易决定它属于哪一区或是否独立

注意：这一步不一定要拆成新的 `.h/.cpp` 文件，先做内部结构分区也可以。

### 5.2 将 `ViewerWindow` 中的“面板同步编排”继续收口

后续可以考虑抽一个轻量同步器或状态适配层，例如：

- `ViewerUiStateAdapter`
- 或 `ViewerControlBinder`

职责：

- 从 `Viewport` 读状态
- 统一回填到 `ViewerControlPanel`
- 集中处理 enable/disable 与显示规则

这样 `ViewerWindow` 可以更专注于：

- 窗口布局
- 生命周期
- 顶层对象装配

### 5.3 将“相机 UI 规则”从控件代码中继续剥离

当前 `CameraControlWidget` 内已经开始有较多“模式 -> 控件状态”的规则。

后续如果继续扩展：

- Perspective + Dolly
- Perspective + Lens
- Orthographic + Lens
- 未来可能的 Orthographic + Dolly

这些规则最好不要继续堆在界面控件里。

建议后续单独形成：

- `camera_ui_state_utils`
- 或 `camera_mode_ui_policy`

用于输出：

- 哪些控件是 primary
- 哪些是 secondary
- 哪些是 inactive

## 6. 当前阶段的结论

当前 UI 层结构：

- 不需要大拆
- 需要继续小步拆解

优先级建议：

1. 先拆相机面板内部职责
2. 再收口 `ViewerWindow` 的同步/编排职责
3. 最后再决定是否引入更明确的 UI policy 层

## 7. 对下一阶段开发的建议

如果进入后续版本，建议按下面顺序推进：

1. 先做 `Zoom Mode` UI 规则整理
- 不一定先做视觉重构
- 先把规则从控件代码中收口

2. 再评审 `Viewport Zoom`
- 明确其不属于 `Camera Zoom UI`

3. 再评审“模型参数变化后的视角保持/自动适配策略”
- 这属于视图响应策略，不应直接并入当前相机主面板

## 8. 讨论结论

- 当前 UI 层已经完成第一轮拆解
- 现阶段需要的是“第二轮职责收紧”，而不是整体重构
- 最优先应继续拆解的是 `CameraControlWidget` 和 `ViewerWindow` 的同步编排职责
- 不建议把 `Viewport Zoom` 和模型参数变化策略直接继续塞进当前相机面板
