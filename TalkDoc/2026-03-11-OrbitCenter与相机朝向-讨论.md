# OrbitCenter与相机朝向讨论

## 1. 讨论背景

在推进 `0.0.6` 过程中，围绕 `orbitCenter` 的显式化与后续聚焦能力扩展，需要先明确一个基础设计问题：

- 当 `Orbit Center` 发生变化时，是否应该考虑当前相机朝向
- 不同类型的操作，是否应采用统一规则

该问题会直接影响后续：

- `Focus Point`
- `Focus All`
- `Focus Selected`
- 平移与缩放交互语义

## 2. 讨论内容

讨论的核心点包括：

1. `setOrbitCenter(worldPoint)` 是否需要考虑当前相机朝向
2. 右键平移 `pan(...)` 是否必须考虑当前相机朝向
3. 基于屏幕交互设置中心时，是否必须依赖当前相机朝向
4. 后续聚焦接口中，“中心是什么”和“相机怎么摆位”是否应该拆开

讨论中形成的关键判断：

- 直接设置旋转中心时，不应引入当前相机朝向对中心定义的干扰
- 通过交互改变中心时，部分操作必须考虑当前相机朝向
- `orbitCenter` 应始终被视为世界空间中的目标点
- 相机朝向影响的是交互映射和相机摆位，而不是中心点本身的语义

## 3. 讨论结论

最终结论如下：

1. `orbitCenter` 统一定义为世界空间目标点
2. `setOrbitCenter(...)` 本身不需要考虑当前相机朝向
3. `pan(...)` 这类基于视口交互的中心变化，必须考虑当前相机朝向
4. 后续接口应区分两类语义：
   - 世界空间语义接口：
     - `setOrbitCenter(worldPoint)`
     - `focusOnPoint(worldPoint)`
     - `focusOnBounds(bounds)`
   - 交互语义接口：
     - `pan(screenDelta)`
     - `rotate(screenDelta)`
     - `zoom(delta)`
5. 后续聚焦功能中，朝向影响相机位置、距离和裁剪策略，但不改变中心点本身的定义

## 4. 对当前版本的影响

对当前 `0.0.6` 的影响如下：

- 可继续推进 `orbitCenter` 的显式化
- `Focus Point / Focus All` 的最小接口可以按世界空间中心语义设计
- 不需要在当前步骤内增加“根据相机朝向修正 orbitCenter”的逻辑

## 5. 后续建议

建议在后续版本中坚持以下设计约束：

1. `orbitCenter` 一律使用世界空间
2. 平移等交互操作继续依赖当前相机朝向做映射
3. 聚焦接口优先面向世界空间点或范围对象，而不是屏幕空间点
4. 若后续引入拾取，再由拾取系统负责“屏幕 -> 世界”的转换
