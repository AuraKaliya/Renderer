# 2026-04-02-0.0.18最小建模算子栈范围-讨论

## 1. 讨论背景

`0.0.17` 已完成参数化建模第一阶段能力：

- 为基础 primitive 建立了显式参数描述
- 建立了统一的 `PrimitiveDescriptor -> MeshData` 求值入口
- 在 `test_viewer` 中提供了 Box / Cylinder / Sphere 的最小参数编辑入口
- 打通了：
  - 参数变化
  - mesh 重建
  - bounds 更新
  - GPU mesh 更新
  - `keep_view / auto_frame`
  闭环

因此后续参数化建模的自然下一步，不再是“继续加 primitive”，而是引入最小建模算子栈。

## 2. 为什么 0.0.18 不应直接做完整 Modifier Stack

虽然当前方向参考了 Blender Modifier Stack 与 Maya Construction History，但现阶段项目仍不具备以下承载条件：

- 选中系统
- 对象树 / 对象级编辑器
- 栈排序 UI
- 节点图
- 多对象依赖关系
- 复杂拓扑编辑能力

因此 `0.0.18` 不适合直接追求“完整 modifier/history 系统”，而应先收口为：

- 固定顺序
- 少量算子
- 单对象求值
- 最小 UI 验证

## 3. 0.0.18 的推荐目标

推荐将 `0.0.18` 定义为：

“在 `0.0.17` 的 primitive 参数对象化基础上，引入最小建模算子栈，使单个参数化对象可以由基础 primitive 与少量算子共同求值。”

核心目标不是“算子数量”，而是建立：

- 对象级参数描述
- 算子描述
- 顺序求值
- 与已有重建/视图策略链路的统一接口

## 4. 推荐协议结构

建议将当前结构从：

`PrimitiveDescriptor -> MeshData`

升级为：

`ParametricObjectDescriptor -> MeshData`

其中：

- `ParametricObjectDescriptor`
  - `PrimitiveDescriptor basePrimitive`
  - `std::vector<OperatorDescriptor> operators`

- `OperatorDescriptor`
  - `OperatorKind kind`
  - `enabled`
  - 对应参数体

建议首批参数体包括：

- `MirrorOperatorSpec`
- `LinearArrayOperatorSpec`
- 可选：
  - `RadialArrayOperatorSpec`

## 5. 为什么首批建议 Mirror + Linear Array

### 5.1 Mirror

优点：

- 参数简单
- 几何规则明确
- 很适合作为第一个“mesh 后处理算子”
- 能验证：
  - enable/disable
  - 轴向选择
  - bounds 变化

### 5.2 Linear Array

优点：

- 适合验证“重复生成 + 合并 mesh”的求值模式
- 参数与 UI 都很直观：
  - count
  - offset
- 与参数化建模场景高度相关

### 5.3 暂不优先 Bevel

原因：

- `Bevel` 不是简单重复或镜像，而是拓扑修改
- 会快速引入：
  - 顶点连接复杂度
  - 法线处理
  - UV 连续性问题
  - 边界稳定性问题

对于当前项目阶段，`Bevel` 更适合在 `0.0.19+` 再讨论。

## 6. 推荐求值路径

建议 `parametric_model` 中的新求值接口为：

`ParametricObjectDescriptor -> build() -> MeshData`

内部流程建议为：

1. 先求值 `basePrimitive`
2. 按固定顺序依次应用 operators
3. 输出最终 `MeshData`
4. 重新计算最终 `localBounds`

当前阶段不要求：

- 算子之间可重排
- 中间缓存
- 局部脏标记
- 增量求值

## 7. test_viewer 的推荐 UI 范围

`0.0.18` 的 UI 不建议做成复杂栈编辑器。

建议最小方案：

- 为当前对象增加一个“Operators”区域
- 固定显示少量算子
- 每个算子提供：
  - `Enabled`
  - 最小参数输入

例如：

Mirror：
- `Enabled`
- `Axis`

Linear Array：
- `Enabled`
- `Count`
- `Offset X`
- `Offset Y`
- `Offset Z`

这样足以验证：

- 算子存在
- 顺序求值
- 与 mesh/bounds/view 更新链路联通

## 8. 与 0.0.16 / 0.0.17 的关系

### 8.1 与 0.0.16 的关系

算子变化属于模型变化，因此：

- operator 参数变化
- operator enable/disable

都必须接入：

- `keep_view`
- `auto_frame`

### 8.2 与 0.0.17 的关系

`0.0.17` 建立了“primitive 参数对象化”，`0.0.18` 则是在这个基础上继续增加：

- 对象级 descriptor
- operator descriptor
- 顺序求值

因此 `0.0.18` 是 `0.0.17` 的直接上层，而不是独立方向。

## 9. 讨论结论

- `0.0.18` 应明确收口为“最小建模算子栈第一版”
- 当前不应直接做完整 Modifier Stack / Construction History / Node Graph
- 推荐优先实现：
  - `Mirror`
  - `Linear Array`
- `Radial Array` 可作为可选项，不应压垮当前版本
- `Bevel` 不建议在 `0.0.18` 首版纳入
- UI 采用固定顺序、最小参数面板即可

## 10. 对版本规划的影响

建议 `0.0.18` 的开发计划按以下三步展开：

1. 建立 `ParametricObjectDescriptor + OperatorDescriptor` 与统一求值入口
2. 实现 `Mirror + Linear Array` 并接入 mesh / bounds / view 闭环
3. 在 `test_viewer` 中补最小 operator UI，完成本版收口
