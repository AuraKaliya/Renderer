# 控制面板滚动与窗口高度策略讨论

## 1. 讨论背景

当前 `test_viewer` 窗口采用：

- 左侧 `Viewport`
- 右侧 `ViewerControlPanel`

随着相机控制项、聚焦项、Bounds 展示项持续增加，当前右侧控制面板已经出现两个明显问题：

1. 整体窗口高度观感偏高
2. 控制参数子界面缺少滚动承载能力

## 2. 当前问题判断

### 2.1 “窗口过高”本质上不是单纯高度值问题

当前主窗口高度由：

- 初始 `resize(1280, 720)`
- 右侧控制面板的最小尺寸与内容自然撑开

共同决定。

如果仅简单缩小主窗口高度，而不处理控制面板承载方式，会导致：

- 下方控件被压缩
- 控件组之间布局拥挤
- 后续继续扩展时仍然失控

所以真正需要调整的不是“只改窗口高度”，而是：

- 主窗口默认高度策略
- 控制面板内容承载方式

### 2.2 控制参数子界面需要滚动容器

当前 `ViewerControlPanel` 内部直接使用竖向布局叠加：

- SceneObjectControlWidget x 3
- LightingControlWidget
- CameraControlWidget
- Bounds
- Actions

这对早期最小验证足够，但现在继续增长已经不适合再靠固定高度硬撑。

因此更合理的承载方式是：

- 保持右侧控制面板外层宽度固定
- 内部改为 `QScrollArea`
- 面板内容在 scroll area 中纵向布局

## 3. 建议方案

### 3.1 主窗口高度策略

建议：

- 维持主窗口可调大小
- 但调低默认初始高度

不建议：

- 将整个窗口高度彻底锁死不可调整

原因：

- 后续 `Bounds`、相机参数、调试信息还会继续增长
- 完全锁高会直接牺牲可用性

更合理的是：

- 设置更克制的默认高度
- 把右侧面板做成可滚动

### 3.2 控制面板滚动策略

建议：

- `ViewerControlPanel` 外部保持现有固定宽度
- 面板内容包装到 `QScrollArea`
- `QScrollArea` 内放一个内容容器 `QWidget`
- 所有子面板继续放在这个内容容器里

这样可以保证：

- 主窗口高度可以更紧凑
- 参数面板不会因为新增控件无限拉高窗口
- 后续继续扩展相机/策略面板时无需推翻结构

### 3.3 Bounds 区域的处理

当前 `Bounds` 属于只读信息区。

建议：

- 保持在滚动面板中
- 不单独拆成独立窗口

因为当前它仍然是视图调试信息的一部分，而不是高频主交互区。

## 4. 对当前结构的影响判断

这项调整不会破坏当前 UI 分层：

- `ViewerWindow`
- `ViewerControlPanel`
- `SceneObjectControlWidget`
- `LightingControlWidget`
- `CameraControlWidget`

它影响的是：

- `ViewerWindow` 的默认尺寸策略
- `ViewerControlPanel` 的内容承载方式

所以这属于“UI 承载结构优化”，不是架构重构。

## 5. 建议的实现边界

如果进入后续版本，建议只做：

1. 下调主窗口默认高度
2. 给 `ViewerControlPanel` 增加 `QScrollArea`
3. 保持现有子面板结构不变

建议暂不做：

1. 每个子面板单独折叠/展开
2. 多列布局
3. 浮动 Dock 化

## 6. 讨论结论

- 当前“窗口过高”的根因是控制面板内容承载方式已到上限
- 更合理的解决方案不是锁死整窗高度，而是：
  - 调低默认窗口高度
  - 让右侧控制面板改为可滚动承载
- 这项需求适合作为下一版本的 UI 承载优化输入，而不应插入当前 `0.0.13` 已冻结的执行计划
