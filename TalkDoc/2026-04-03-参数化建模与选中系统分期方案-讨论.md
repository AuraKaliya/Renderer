# 参数化建模与选中系统分期方案讨论

## 1. 讨论背景

上一轮讨论已经明确：

- 后续参数化建模不应优先走 raw mesh 顶点编辑
- 应优先建立：
  - 对象层
  - 特征层
  - 控制点层
- 选中系统应围绕：
  - `Object Mode`
  - `Feature Mode`
  - `Control Point Mode`
  逐步展开

但要真正进入开发，还需要进一步回答：

1. 哪些能力应先做
2. 哪些能力不应混入同一版本
3. 每一阶段做完后，测试界面可以验证什么

## 2. 总体分期结论

建议把后续能力拆成四个版本阶段推进：

1. `0.0.20`
- 建立对象级参数化编辑模型

2. `0.0.21`
- 建立最小选中系统与对象/特征编辑闭环

3. `0.0.22`
- 建立控制点模型与最小控制点编辑闭环

4. `0.0.23+`
- 继续扩展曲线/剖面驱动建模与更正式的 gizmo / 拾取能力

这个顺序的关键点是：

- 先把数据模型建对
- 再把选中状态建对
- 最后再让视口直接编辑

## 3. 推荐分期

### 3.1 `0.0.20`：对象级参数化编辑模型

#### 目标

把当前“固定三个对象 + 各自 descriptor”的测试性结构，升级为更正式的参数化对象模型。

#### 本版建议包含

- 正式引入对象级建模结构，例如：
  - `ParametricObjectId`
  - `ParametricFeatureId`
  - `ParametricObjectDescriptor`
  - `FeatureDescriptor`
- 将当前：
  - `basePrimitive`
  - `operators`
  统一收口到“对象 + 特征链”的结构中
- 为对象增加稳定元数据：
  - `name`
  - `object kind`
  - `local transform`
  - `pivot/origin`
- 为特征增加稳定元数据：
  - `feature kind`
  - `enabled`
  - `local frame`
  - `feature name`

#### 本版明确不包含

- 视口拾取
- 控制点拖拽
- gizmo
- mesh component 选择
- 多对象复杂编辑

#### 本版完成后应能验证

- 一个对象不再只是“primitive + operators”的临时结构，而是正式对象
- 当前对象拥有可稳定引用的对象 id 和特征 id
- UI 可按对象和特征显示结构，而不是只按固定卡片展示

#### 为什么必须先做这一版

如果没有正式的对象 / 特征层，后续“选中谁、删除谁、编辑谁、控制点属于谁”都没有稳定语义。

这一步是所有后续编辑能力的基础。

### 3.2 `0.0.21`：最小选中系统与对象/特征编辑闭环

#### 目标

在不引入复杂屏幕拾取的前提下，先把对象与特征的选中、激活、编辑、添加、删除闭环跑通。

#### 本版建议包含

- 新增 `SelectionState`
- 显式区分：
  - `selected object ids`
  - `active object id`
  - `selected feature ids`
  - `active feature id`
- 在测试界面中提供最小选中入口：
  - 对象列表
  - 当前对象的特征列表
  - 激活对象 / 激活特征
- 支持：
  - 添加 feature
  - 删除 feature
  - 启用 / 禁用 feature
- 让 `Focus Selected` 或“以当前对象为中心”进入最小闭环

#### 本版明确不包含

- 屏幕空间拾取
- 框选
- Control Point Mode
- 控制点反推参数
- 复杂多选策略

#### 本版完成后应能验证

- 当前对象可被显式选中和激活
- 当前特征可被显式选中和激活
- 当前特征可以：
  - 添加
  - 删除
  - 启用 / 禁用
- 当前相机和聚焦能力可以和“当前选中对象”对齐

#### 为什么这一步应早于控制点编辑

控制点永远从属于对象和特征。

如果对象和特征的选择、激活、删除语义还不稳定，直接做控制点拖拽只会把状态搞乱。

### 3.3 `0.0.22`：控制点模型与最小控制点编辑闭环

#### 目标

让参数化建模从“面板编辑参数”进入“视口中直接编辑参数”的第一阶段。

#### 本版建议包含

- 正式引入 `ControlPointDescriptor`
- 为特征声明控制点集合
- 首批只支持最简单、反推关系最清晰的控制点：
  - Box 尺寸控制点
  - Cylinder 半径 / 高度控制点
  - Mirror 平面位置点
  - Linear Array 方向/偏移点
- 支持：
  - 控制点显示
  - 控制点选中
  - 控制点拖拽
  - 控制点反推参数
- 引入最小坐标空间模型：
  - `world`
  - `object_local`
  - `feature_local`

#### 本版明确不包含

- 通用 gizmo 系统
- 曲线 CV 编辑
- 软选择
- mesh component mode
- 复杂约束系统

#### 本版完成后应能验证

- 当前激活对象的部分特征能显示控制点
- 控制点拖拽后会稳定驱动参数更新
- 参数更新仍走统一路径：
  - feature param change
  - rebuild
  - bounds update
  - view strategy update

#### 为什么这一步只做最小闭环

控制点编辑一旦范围放大，很容易演变成：

- 通用 gizmo
- 通用约束
- 通用拾取
- 通用 component 编辑

这会远超当前项目可控范围。

### 3.4 `0.0.23+`：曲线/剖面与更正式的编辑器能力

#### 目标

在对象、特征、选中、控制点都稳定后，再进入更接近经典建模软件的高级参数建模能力。

#### 后续建议方向

- 曲线 / 剖面驱动建模
  - path + profile
  - extrude
  - revolve
  - sweep
- 更正式的控制点类型
  - curve CV
  - tangent handle
  - profile anchor
- 更正式的视口编辑能力
  - gizmo
  - screen-space picking
  - multi-selection
  - transform orientation

## 4. 单元分类的落地建议

建议把运行时单元先稳定为四类：

1. `Object Unit`
- 场景里的参数化对象

2. `Feature Unit`
- 对象内部的一个建模步骤

3. `Control Unit`
- 驱动 feature 参数的控制点或控制柄

4. `Result Unit`
- 求值后的 mesh / bounds / preview

优先级建议是：

- 先建 `Object Unit`
- 再建 `Feature Unit`
- 再建 `Control Unit`
- `Result Unit` 始终只是输出，不作为早期主要编辑对象

## 5. 选中系统的分层建议

### 5.1 必须先有“激活”概念

建议从一开始就明确：

- `selected`
- `active`

原因是 Blender 和 Maya 都存在“被选中集合”和“当前操作目标”的区别。

如果不先区分，后续会出现：

- 选中了多个对象，但编辑面板不知道编辑谁
- 选中了多个特征，但控制点不知道挂到谁上

### 5.2 首批不必强推复杂多选

建议第一阶段只保证：

- 单选稳定
- `Shift` 多选可留到后续
- active target 始终唯一

### 5.3 选中目标的优先顺序

建议后续拾取时的优先顺序是：

1. gizmo / control handle
2. control point
3. feature helper
4. object

这样更符合经典建模软件的编辑体验。

但这一顺序不需要在最早版本就全部实现，只需要作为后续设计约束保留。

## 6. 当前项目最不该过早做的内容

本次讨论明确认为，以下内容不适合作为下一版本主目标：

1. 完整屏幕拾取系统
2. 通用 mesh 顶点/边/面编辑
3. 通用 gizmo 框架
4. 曲线建模 + 控制点 + 历史链一次性一起上
5. 节点图编辑器

原因很简单：

- 当前项目刚完成参数对象化和测试界面收口
- 如果下一步直接跨入通用编辑器能力，项目复杂度会突然抬升
- 更稳妥的路径是先把“对象 / 特征 / 选中”建模做扎实

## 7. 推荐的下一版切入点

如果基于当前仓库继续推进，最推荐的下一版目标是：

### 推荐 `0.0.20`

主题：

- 参数化对象正式对象化 + 特征链结构显式化

最小闭环：

- 一个场景中可以有多个正式 `ParametricObject`
- 每个对象包含：
  - 基础生成特征
  - 若干修改特征
- UI 中可展示对象列表和特征列表
- 仍使用当前测试界面进行最小验证

不建议把 `0.0.20` 直接做成：

- 选中拾取版
- 控制点拖拽版
- gizmo 版

## 8. 讨论结论

- 下一阶段的核心不是“继续加 primitive”，而是把参数化对象升级成真正可编辑的对象模型
- 推荐的能力推进顺序是：
  1. 正式对象模型
  2. 最小选中系统
  3. 最小控制点编辑
  4. 更正式的视口编辑与曲线建模
- 这条路线既能和 Blender / Maya 的成熟思路对齐，又不会一步跨进完整 DCC 编辑器复杂度
