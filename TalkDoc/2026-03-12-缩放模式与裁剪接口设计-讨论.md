# 缩放模式与裁剪接口设计讨论

## 1. 讨论背景

在确认当前轨道相机继续以 Dolly Zoom 作为主缩放模式后，进一步引出了两个设计问题：

- 若滚轮缩放过程中需要联动更新 `near/far`，是否应拆解当前 `near/far` 的计算逻辑
- 若后续支持正交 / 透视模式切换，并允许多种 Zoom 模式切换，应如何设计接口边界

## 2. 讨论内容

本次讨论聚焦以下问题：

1. `near/far` 的计算逻辑是否应继续内嵌在现有聚焦 helper 中
2. 正交 / 透视模式切换后，Zoom 模式是否应独立建模
3. 不同投影模式下，Dolly Zoom 与 Lens Zoom 的职责边界应如何定义

## 3. 讨论结论

### 3.1 应拆解 near/far 计算逻辑

- 当前 `near/far` 计算已同时被：
  - `Focus All`
  - `Focus Point`
  - 后续拟加入的滚轮缩放
  所依赖
- 若继续把 `near/far` 逻辑嵌在聚焦 helper 内，后续会出现：
  - 逻辑重复
  - 调整一处影响多处但边界不清
  - 难以支持不同投影模式

因此，建议将裁剪计算拆为独立策略函数。

### 3.2 投影模式与缩放模式应分离

- `ProjectionMode` 应只描述：
  - `Perspective`
  - `Orthographic`
- `ZoomMode` 应独立描述：
  - `Dolly`
  - `Lens`
- 两者不应混成一个枚举，否则后续组合会失控

### 3.3 正交模式下的默认缩放

- 从纯数学与常见编辑器习惯看，正交模式下的主缩放应优先理解为“视口缩放 / 正交尺寸缩放”
- 也就是更接近 Lens Zoom，而不是 Dolly
- 但如果项目明确希望在正交模式下默认仍使用 Dolly，也可以支持
- 关键不是“能不能”，而是接口必须允许：
  - 投影模式切换
  - 缩放模式切换
  - 每种投影模式拥有自己的默认缩放模式

## 4. 推荐接口设计

### 4.1 枚举层

- `enum class ProjectionMode { Perspective, Orthographic }`
- `enum class ZoomMode { Dolly, Lens }`

### 4.2 相机控制器状态层

建议 `OrbitCameraController` 后续具备：

- `setProjectionMode(...)`
- `projectionMode()`
- `setZoomMode(...)`
- `zoomMode()`
- `setZoomModeForProjection(projectionMode, zoomMode)`

### 4.3 裁剪与缩放计算拆层

建议新增独立计算模块，例如：

- `camera_projection_utils`
- `camera_clip_utils`

其中：

- `camera_focus_utils`
  - 负责聚焦目标和 framing
- `camera_clip_utils`
  - 负责 `near/far` 计算
- `camera_zoom_utils`
  - 负责不同 `ZoomMode` 下的缩放更新

### 4.4 正交模式参数

若引入正交模式，建议不要复用 `verticalFovDegrees` 语义，而是新增：

- `orthographicHeight`

这样可以避免：

- 透视下用 `FOV`
- 正交下仍拿 `FOV` 做无意义占位

## 5. 对后续版本规划的影响

- 这次讨论不应插入当前 `0.0.7`
- 应作为下一版本评审输入
- 下一版若要做缩放裁剪稳定性和投影模式扩展，建议先完成：
  1. `near/far` 计算拆层
  2. `ProjectionMode / ZoomMode` 状态建模
  3. 仅透视模式下的 Dolly 缩放裁剪联动
  4. 再考虑正交模式接入
