# ANGENT.md

## 1. 目的

本文档定义项目开发过程中的通用版本执行规范。  
该规范可直接迁移到其他项目复用。

## 2. 版本规则

- 开发阶段版本号从 `0.0.1` 开始
- 默认按小步迭代递增
- 若无额外说明，版本号按补丁位递增，例如：
  - `0.0.1`
  - `0.0.2`
  - `0.0.3`
- 若一次迭代范围明显扩大，可在沟通后调整次版本号

## 3. 文档目录规则

项目根目录下必须存在：

- `VersionDoc/`
- `DebugDoc/`
- `ReviewDoc/`
- `TalkDoc/`

目录用途如下：

- `VersionDoc/`：存放每个版本的计划文档和开发记录文档
- `DebugDoc/`：存放开发过程中已定位、已解决、可复用的问题记录文档
- `ReviewDoc/`：存放开发过程中发现但未纳入当前版本处理的评审遗留项文档
- `TalkDoc/`：存放开发过程中讨论内容和讨论结果的备份文档

## 4. 开发前规则

在开始某一个版本的实际开发前，必须先在 `VersionDoc/` 中生成一份计划文档。

命名规则：

- `【版本号】-开发计划.md`

示例：

- `0.0.1-开发计划.md`
- `0.0.2-开发计划.md`

计划文档至少应包含：

- 版本目标
- 范围说明
- 分步骤开发内容
- 每一步的完成判定
- 测试与构建责任说明
- 版本完成后的收尾要求

## 5. 开发执行规则

- 开发按计划文档中的步骤顺序执行
- 每完成一个步骤后，暂停并等待用户进行构建与测试
- 构建和测试默认由用户执行
- 用户明确确认当前步骤通过后，才允许进入下一步
- 若当前步骤未通过，优先修复当前步骤问题，不得跳步

### 5.1 计划冻结规则

- 一个版本的 `开发计划.md` 一旦进入执行，即视为冻结
- 执行过程中不得直接改动当前版本既有范围、步骤和目标
- 新讨论出的优化点、补充点、结构整理点，不得直接插入当前版本
- 若确需调整计划，必须先重新完成架构评审和功能评审，再更新计划文档后执行

### 5.2 当前版本允许直接处理的例外

当前版本执行中，只允许直接插入处理以下内容：

- 当前步骤的编译错误
- 当前步骤的运行错误
- 当前步骤的功能偏差
- 会导致当前步骤目标失真的阻塞性缺陷

以下内容不得直接插入当前版本：

- 新工具函数补全
- 非当前步骤必须的抽象整理
- 未来步骤才会使用的扩展接口
- 纯结构优化
- 非阻塞性的设计补强

### 5.3 开发中发现项分类规则

开发过程中发现的新事项，必须先分类，再决定处理方式：

#### 5.3.1 阻塞问题

满足以下任一条件的，视为阻塞问题：

- 当前步骤无法继续
- 当前步骤输出错误
- 当前步骤验证目标无法达成

处理规则：

- 必须在当前步骤内修复
- 修复完成并经用户验证后，可按问题处理流程归档到 `DebugDoc/`

#### 5.3.2 非阻塞疏漏

满足以下特征的，视为非阻塞疏漏：

- 当前步骤仍可正确完成
- 属于设计补全、语义完善、结构优化、工具函数补齐、未来扩展准备
- 不修复不会导致当前步骤结果错误

处理规则：

- 不得直接插入当前版本实现
- 必须登记到 `ReviewDoc/`
- 待后续架构评审和功能评审后，再决定是否纳入新版本

## 6. 开发完成规则

当某一个版本计划中的全部步骤完成后，必须在 `VersionDoc/` 中生成一份开发记录文档。

命名规则：

- `【版本号】-开发记录.md`

示例：

- `0.0.1-开发记录.md`
- `0.0.2-开发记录.md`

开发记录文档至少应包含：

- 版本目标
- 实际完成内容
- 涉及文件
- 用户构建/测试结论
- 本版本问题修复
- 已知问题与限制
- 本版本遗留项
- 下一版本建议

当开发记录文档完成且用户确认当前版本通过后，必须同步进行一次 git 版本提交。

git 提交要求：

- 每个版本结束时进行一次提交
- 提交内容应覆盖该版本实际完成的开发变更
- 提交说明允许使用多行 message
- 第一行必须简要概括当前版本的核心改动
- 后续各行用于列出具体改动点、功能点或结构调整点

提交 message 建议格式：

```text
0.0.X: 简要概括当前版本改动

- 改动点 1
- 改动点 2
- 改动点 3
```

## 7. 问题处理与归档规则

当开发过程中出现由用户指出的问题，并且该问题已经完成定位、修复、验证并准备归档时，必须在 `DebugDoc/` 目录下生成一份问题记录文档。

命名规则：

- `【版本号】-问题.md`

示例：

- `0.0.1-问题.md`
- `0.0.2-问题.md`

问题记录文档至少应包含以下内容：

- `问题描述`
- `问题原因`
- `解决方案`
- `期望结果`

编写要求：

- 问题描述应准确描述用户反馈的现象
- 问题原因应说明根因，而不是只描述表象
- 解决方案应记录可复用的修复思路和关键修改点
- 期望结果应明确修复后应表现出的正确行为

归档时机：

- 问题已完成修复
- 用户已完成构建和测试
- 用户确认问题关闭或可以归档

复用目标：

- `DebugDoc/` 中的问题文档应可作为后续同类问题的排查与修复参考
- 若后续项目出现相似问题，应优先查阅已有问题记录

## 8. 评审遗留项登记规则

当开发过程中发现非阻塞疏漏，且该事项不应直接插入当前版本时，必须在 `ReviewDoc/` 下登记评审遗留项文档。

命名规则：

- `【版本号】-评审遗留项.md`

示例：

- `0.0.5-评审遗留项.md`
- `0.0.6-评审遗留项.md`

评审遗留项文档至少应包含：

- `背景`
- `发现时间`
- `所属版本`
- `是否阻塞当前步骤`
- `风险说明`
- `建议处理版本`
- `建议方案`

使用规则：

- 评审遗留项用于沉淀本版本中发现但未纳入处理的设计问题、语义缺口、优化建议
- 未经过评审确认的遗留项，不得直接进入后续开发实现
- 后续版本计划应明确说明其是否吸收了某份遗留项文档中的内容

## 9. 讨论记录规则

当开发过程中发生架构讨论、功能讨论、设计取舍讨论时，必须将讨论内容和讨论结果整理为文档，并存放在 `TalkDoc/` 中备份。

命名规则：

- `【日期】-【主题】-讨论.md`

示例：

- `2026-03-11-OrbitCenter与相机朝向-讨论.md`
- `2026-03-12-裁剪策略-讨论.md`

讨论文档至少应包含：

- `讨论背景`
- `讨论内容`
- `讨论结论`
- `对当前版本的影响`
- `后续建议`

使用规则：

- 讨论一旦形成明确结论，应及时归档
- 讨论结论如影响后续版本规划，应作为 `VersionDoc/` 或 `ReviewDoc/` 的输入来源
- 若讨论结论被采纳为设计约束，后续评审文档和开发计划应与该结论保持一致

## 10. 变更范围规则

- 若开发过程中用户调整范围，需先更新对应版本的开发计划文档
- 计划更新完成后，才继续执行新的开发内容
- 不允许计划外功能直接混入当前版本

## 11. 下一版本计划输入规则

在创建新版本 `开发计划.md` 前，必须先检查以下输入来源：

- 最新的评审结论文档
- `ReviewDoc/` 中与当前目标相关的遗留项文档
- `DebugDoc/` 中最近归档的问题文档
- `TalkDoc/` 中与当前目标直接相关的讨论文档
- 上一版本 `开发记录.md` 中的“本版本遗留项”

新版本计划应明确说明其来源基础，确保开发计划来自已审查的输入，而不是执行过程中临时插入的新想法。

## 12. 角色分工规则

默认分工如下：

### 12.1 Codex

- 负责方案拆分
- 负责代码实现
- 负责版本计划文档编写
- 负责版本开发记录文档编写
- 负责评审遗留项文档编写
- 负责讨论文档整理与归档
- 负责在版本完成并获用户确认后执行对应的 git 提交
- 负责在每一步完成后明确告知当前状态

### 12.2 用户

- 负责构建
- 负责运行测试
- 负责验收当前步骤是否通过
- 负责决定是否进入下一步骤

## 13. 默认工作节奏

每个版本遵循以下流程：

1. 创建 `【版本号】-开发计划.md`
2. 执行步骤 1
3. 用户构建和测试
4. 用户确认通过
5. 执行步骤 2
6. 用户构建和测试
7. 用户确认通过
8. 直至该版本计划全部完成
9. 创建 `【版本号】-开发记录.md`
10. 执行一次 git 提交，提交当前版本全部变更

若中途出现问题并完成修复闭环，则补充：

1. 在 `DebugDoc/` 下创建 `【版本号】-问题.md`
2. 记录问题与解决方案
3. 经用户确认后归档

若中途发现非阻塞疏漏，则补充：

1. 在 `ReviewDoc/` 下创建或更新 `【版本号】-评审遗留项.md`
2. 记录疏漏背景、风险和建议处理版本
3. 当前版本不直接插入处理
4. 在下一版本规划前作为评审输入重新审查

若中途发生明确的讨论结论，则补充：

1. 在 `TalkDoc/` 下创建 `【日期】-【主题】-讨论.md`
2. 记录讨论内容和结论
3. 若结论影响当前或后续版本，纳入对应文档输入链

## 14. 适用说明

本规范适用于需要分阶段开发、分步骤验收、由用户负责构建与测试的项目。  
迁移到其他项目时，保留本文件即可沿用同样流程。
