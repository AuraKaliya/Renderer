# ANGENT.md

## 1. 目的

本文档定义项目开发过程中的通用版本执行规范。  
该规范可直接迁移到其他项目复用。

## 2. 版本规则

- 开发阶段版本号从 `0.0.1` 开始
- 默认按小步迭代递增
- 若无额外说明，版本号按补丁位递增，例如：
  - `0.0.1`
  - `0.0.2`
  - `0.0.3`
- 若一次迭代范围明显扩大，可在沟通后调整次版本号

## 3. 文档目录规则

项目根目录下必须存在：

- `VersionDoc/`
- `DebugDoc/`

目录用途如下：

- `VersionDoc/`：存放每个版本的计划文档和开发记录文档
- `DebugDoc/`：存放开发过程中已定位、已解决、可复用的问题记录文档

## 4. 开发前规则

在开始某一个版本的实际开发前，必须先在 `VersionDoc/` 中生成一份计划文档。

命名规则：

- `【版本号】-开发计划.md`

示例：

- `0.0.1-开发计划.md`
- `0.0.2-开发计划.md`

计划文档至少应包含：

- 版本目标
- 范围说明
- 分步骤开发内容
- 每一步的完成判定
- 测试与构建责任说明
- 版本完成后的收尾要求

## 5. 开发执行规则

- 开发按计划文档中的步骤顺序执行
- 每完成一个步骤后，暂停并等待用户进行构建与测试
- 构建和测试默认由用户执行
- 用户明确确认当前步骤通过后，才允许进入下一步
- 若当前步骤未通过，优先修复当前步骤问题，不得跳步

## 6. 开发完成规则

当某一个版本计划中的全部步骤完成后，必须在 `VersionDoc/` 中生成一份开发记录文档。

命名规则：

- `【版本号】-开发记录.md`

示例：

- `0.0.1-开发记录.md`
- `0.0.2-开发记录.md`

开发记录文档至少应包含：

- 版本目标
- 实际完成内容
- 涉及文件
- 用户构建/测试结论
- 已知问题与限制
- 下一版本建议

当开发记录文档完成且用户确认当前版本通过后，必须同步进行一次 git 版本提交。

git 提交要求：

- 每个版本结束时进行一次提交
- 提交内容应覆盖该版本实际完成的开发变更
- 提交说明允许使用多行 message
- 第一行必须简要概括当前版本的核心改动
- 后续各行用于列出具体改动点、功能点或结构调整点

提交 message 建议格式：

```text
0.0.X: 简要概括当前版本改动

- 改动点 1
- 改动点 2
- 改动点 3
```

## 7. 问题处理与归档规则

当开发过程中出现由用户指出的问题，并且该问题已经完成定位、修复、验证并准备归档时，必须在 `DebugDoc/` 目录下生成一份问题记录文档。

命名规则：

- `【版本号】-问题.md`

示例：

- `0.0.1-问题.md`
- `0.0.2-问题.md`

问题记录文档至少应包含以下内容：

- `问题描述`
- `问题原因`
- `解决方案`
- `期望结果`

编写要求：

- 问题描述应准确描述用户反馈的现象
- 问题原因应说明根因，而不是只描述表象
- 解决方案应记录可复用的修复思路和关键修改点
- 期望结果应明确修复后应表现出的正确行为

归档时机：

- 问题已完成修复
- 用户已完成构建和测试
- 用户确认问题关闭或可以归档

复用目标：

- `DebugDoc/` 中的问题文档应可作为后续同类问题的排查与修复参考
- 若后续项目出现相似问题，应优先查阅已有问题记录

## 8. 变更范围规则

- 若开发过程中用户调整范围，需先更新对应版本的开发计划文档
- 计划更新完成后，才继续执行新的开发内容
- 不允许计划外功能直接混入当前版本

## 9. 角色分工规则

默认分工如下：

### 9.1 Codex

- 负责方案拆分
- 负责代码实现
- 负责版本计划文档编写
- 负责版本开发记录文档编写
- 负责在版本完成并获用户确认后执行对应的 git 提交
- 负责在每一步完成后明确告知当前状态

### 9.2 用户

- 负责构建
- 负责运行测试
- 负责验收当前步骤是否通过
- 负责决定是否进入下一步骤

## 10. 默认工作节奏

每个版本遵循以下流程：

1. 创建 `【版本号】-开发计划.md`
2. 执行步骤 1
3. 用户构建和测试
4. 用户确认通过
5. 执行步骤 2
6. 用户构建和测试
7. 用户确认通过
8. 直至该版本计划全部完成
9. 创建 `【版本号】-开发记录.md`
10. 执行一次 git 提交，提交当前版本全部变更

若中途出现问题并完成修复闭环，则补充：

1. 在 `DebugDoc/` 下创建 `【版本号】-问题.md`
2. 记录问题与解决方案
3. 经用户确认后归档

## 11. 适用说明

本规范适用于需要分阶段开发、分步骤验收、由用户负责构建与测试的项目。  
迁移到其他项目时，保留本文件即可沿用同样流程。
